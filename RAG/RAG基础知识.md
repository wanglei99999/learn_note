# 总结

RAG的内容其实不算多，我把rag的思路总结出来：我将rag划分到四个流程
```
索引层：文件->chunk->embedding->入库 这个流程
  |
查询层 用户输入query 预处理流程
  |
召回层 向量召回流程
  |
生成层 召回的topk到llm生成答案的流程
```

每层都有它可以优化的地方：
索引层是入库流程，优化主要是chunk的处理包括：
1. 固定+overlap
2. 语义分割
3. 结构分割：章段句
4. 命题分割
5. 父子分割
6. Contextual Retrieval（Anthropic 2024）
其中，其实父子分割，摘要分割就是小检索 大输出的思想。

查询层的优化主要是：
1. 语义改写
2. step-back改写
3. HyDe文档嵌入
4. 多query改写

召回层的优化主要是多路召回+rerank，已经属于标配了
生成层我的理解是主要是为了评估质量、防止模型幻觉，这里可以思考的优化是：
1. prompt增强
2. 检索质量门控
3. 生成后引用核查
4. 结构化输出溯源

然后是rag评估，核心是**分层打分**，好处是能定位到底是哪一层的锅：
1. 检索层：Hit@K（Top-K 有没有正确 chunk）、MRR（正确 chunk 排第几）——看找没找到、排没排前
2. 生成层：RAGAs 四指标（LLM 当裁判）——忠实度（真不真/幻觉）、相关性（切不切题）、上下文召回（够不够）、上下文精确（准不准）
3. 线上：踩率、转人工率、会话解决率兜底

指标低就反查优化方向：Hit@K 低→换 Embedding，MRR 低→加 Rerank，忠实度低→Prompt 约束+引用核查。指标不是为了打分，是为了定位问题。

最后是几种rag形式，native、advanced、modular 三种演变，新 sele rag、graph rag、crag三种新rag优化。
# 一、为什么需要 RAG

## 根源:LLM 的「知识冻结」

LLM 的知识是预训练时从海量文本「编码」进参数权重的,像一本**印刷完成就无法修改的百科全书**(知识烧进了 ROM)。训练完成后:

- 训练截止日期之后的事,它一无所知
- 重新训练成本千万美元级,不可能每周来一遍
- 微调能注入知识,但成本高、无法溯源、出错难定位

这个固有特性导致三个环环相扣的问题:

1. **知识时效性**:问它训练截止后的事,它不会说"不知道",而是用历史规律"推测"一个听起来合理的答案
2. **私有知识空白**:公司内部文档、行业资料压根没进过训练集——时效性是"旧了",私有知识是"压根没有"
3. **幻觉**:LLM 的核心机制是"预测下一个词",没有"我不确定就停下"的开关,知识不够时只能硬着头皮拼凑

>  关键认知:**幻觉不是独立的 bug, 而是知识缺失和模型生成机制的缺陷的副产品**。RAG 是**补全事实输入层**的方案，治的不是幻觉本身,而是幻觉的根源——参数里没有对应知识这一问题。

## RAG 的解法

把知识存到**外部知识库**,用户提问时**实时检索注入 prompt**,LLM 基于"参考资料"作答——给 LLM 开**开卷考试**,不再靠死记硬背。

两大核心价值:
- **知识可热更新**:加文档即生效,不动模型参数,成本极低
- **答案可溯源**:每条回答能追溯到来源 chunk,答错了知道是哪条知识的问题

## RAG vs 微调

本质区别:**微调改模型参数,RAG 不改参数、推理时注入知识**。

| 维度 | 微调(Fine-tuning) | RAG |
| --- | --- | --- |
| 知识更新 | 需重新训练,成本高 | 更新知识库即可,实时生效 |
| 推理延迟 | 低,无额外检索步骤 | 较高,多一次检索耗时 |
| 实现成本 | 高,需 GPU 和标注数据 | 低,向量库 + Embedding 即可 |
| 答案可溯源 | 不支持,来自模型参数 | 支持,可追溯到具体 chunk |
| 适合场景 | 定制输出风格、深度专业能力 | 私有知识问答、动态更新数据 |
| 知识上限 | 受限于训练数据质量和规模 | 受限于检索质量和 context 长度 |

- 微调三痛点:成本高;业务数据每周变、不可能每周重训;不透明(记住了什么、记错了什么无从得知)
- RAG 三短板:多几百 ms 检索延迟;**检索质量卡死上限**(没检索到的内容 LLM 变不出来);对深度推理能力提升有限

> ⚠️ 关键直觉:生成层(LLM)只是在**复述和整理**检索到的内容,它的上限被检索层死死卡住。RAG 调优的主战场永远是检索层,不是换更贵的 LLM。

**不是二选一**:实际工程常组合使用——先微调让模型学会输出格式、语气、行业术语,再套 RAG 提供具体知识。**微调解决「怎么说」,RAG 解决「说什么」**。

# 二、整体架构:离线建库 + 在线问答

```
离线阶段(建库,只做一次):
  文档加载 → 切割 Chunking → 向量化 Embedding → 入库

在线阶段(每次提问实时执行):
  Query 预处理 → Query 向量化 → 多路召回(粗排) → Rerank(精排) → Prompt 拼装 → 生成 + 溯源
```

## 离线阶段四步

1. **文档加载**:PDF/Word/Markdown/网页/数据库,用 LangChain / LlamaIndex 的 DocumentLoader,支持几十种格式
2. **文档切割**:切成 chunk(详见第三章)
3. **向量化**:Embedding 模型把每个 chunk 转成高维向量(详见第四章)
4. **入库**:向量 + 原文 + metadata 存入向量数据库(详见第五章)

## 在线阶段六步

### ① Query 预处理(改写)

用户提问口语化、带指代(「上次说的那个方案怎么样」离开上下文没法检索)。用 LLM 改写成适合检索的独立问题,方法有直接改写、HyDE、多角度扩写(详见第六章·查询层)。

### ② Query 向量化

**必须用和离线建库完全相同的 Embedding 模型。** 不同模型的向量空间"坐标系"不同(维度都可能对不上),A 模型建库、B 模型查询,距离计算完全没有意义。这也是为什么**换 Embedding 模型必须重建整个知识库**。

### ③ 向量检索(粗排)+ 多路召回

用问题向量做 ANN 搜索,取相似度最高的 Top-K。百万级向量库几十毫秒返回。工程上通常同时跑**向量检索(管语义)+ BM25(管精确词)**,用 RRF 融合(详见第六章·召回层)。

### ④ Rerank(精排)

粗排 Top-20 里难免混入"看着近但不相关"的内容。Rerank 模型(Cross-Encoder)把问题和每个候选拼在一起深度判断相关性,重新排序,留下 Top-3~5。**粗排像肉眼扫书架抽书,精排是一本本翻开读目录确认。**

### ⑤ Prompt 拼装

```
你是一个专业助手,请根据以下参考资料回答用户的问题。
如果参考资料中没有相关信息,请回答「根据现有资料无法回答」,不要自行猜测。

参考资料:
[1] {chunk_1}
[2] {chunk_2}
[3] {chunk_3}

用户问题:{user_query}
```

每条指令都有工程意图:「只根据资料回答」抑制凭记忆发挥;「没有就说不知道」防止强行补全出幻觉;**资料带编号**是为了后续引用溯源。

### ⑥ 生成 + 溯源

要求 LLM 标注每句话来自哪个片段(「根据资料[1]…」),用户可回原文核实。即使有参考资料,LLM 仍可能过度发挥或误读,溯源给了用户判断的能力。

## 耗时分布与优化

| 环节 | 耗时 |
| --- | --- |
| Query 改写 | 几十 ms |
| 向量检索 | 几十 ms |
| Rerank | 几百 ms |
| Prompt 组装 | 可忽略 |
| LLM 生成 | 1~10s(取决于输出长度) |

常见优化:缓存高频 Query 的检索结果;向量检索和 BM25 **并行执行**,两路延迟从相加变成取最大值。

# 三、文档切割(Chunking)

## 为什么不能整篇存

1. Embedding 模型有输入长度限制(几百到几千 token),整篇塞不进去
2. 更关键:整篇压成一个向量,细节信息被**「平均掉」**——你想找「退款政策」,向量里混着「配送时效」「积分规则」,检索到的只能是笼统的整篇,不是精确的那段话
![[Pasted image 20260708084859.png]]
## chunk的一些trick

实际上，全文和chunk的思路是反过来的，文段越长，信息越杂，噪音越多，越不利于llm回答问题，chunk的思路是将信息切成更小的段落，这样信息越集中。但也出现了问题，背景相关的上下文会丢失，这也是后面，父子找回、上下文召回的出发点。

## 每条记录的结构:向量 + 原文 + metadata

一篇文档 → 向量库里的**多条记录**(一对多)。每条记录三部分,可类比为:

- **向量 = 索引卡**:标记这段内容在语义空间的位置,只给检索系统用,LLM 看不懂
- **原文 = 书页**:检索命中后原封不动塞进 prompt,LLM 真正阅读的内容
- **metadata = 书签**:来源文件、页码、章节,用于**过滤**(只搜某部门文档)和**溯源**(答案来自哪页)

**向量负责「找到」,原文负责「阅读」,缺一不可。**
![[Pasted image 20260708084940.png]]
## 粒度怎么定

- 太大(如 2000 token):语义杂,向量笼统,召回不精准,还容易超模型输入限制
- 太小(如 50 token):语义不完整,LLM 看不懂,检索噪音多
- **合理起点:500~1000 token**,再按文档类型选策略

## 切割策略对比

| 策略            | 适用文档            | 优点                  | 缺点                 |
| ------------- | --------------- | ------------------- | ------------------ |
| 固定大小 + 重叠     | 纯文本、无结构         | 实现简单、大小可控           | 可能在语义中间截断          |
| 语义边界切割        | 段落分明的文章         | 语义完整、召回质量好          | 实现稍复杂、大小不均         |
| 标题层级切割        | Markdown / HTML | 天然语义独立、带结构 metadata | 依赖清晰标题结构           |
| 代码按函数切割       | 代码文件            | 保留逻辑完整性             | 需 AST 解析、限定语言      |
| 表格整块保留        | 表格              | 保留表头与行列关系           | —                  |
| 父子切割          | 追求高质量的通用场景      | 检索精准 + 上下文完整两全      | 存储翻倍、索引复杂          |
| Late Chunking | 各类文档            | chunk 向量保留全文上下文     | 需长上下文 Embedding 模型 |

要点:
- **语义边界切割**:维护分隔符优先级列表(段落→句子→标点),永不在句中截断;可用 spacy/nltk 识别句子边界
- **代码**:以函数/类为最小单元(AST 解析),函数被拦腰截断就失去逻辑意义
- **表格**:整块转 Markdown 存,不能按行截断——「2 小时」单独看毫无意义,配上表头「响应时间:2 小时」才成立
- **Late Chunking**(Jina AI 2024):**先编码后切分**——整篇过长上下文 Embedding 模型,拿到 token 级向量(已通过注意力互相感知),再按 chunk 边界做 mean pooling。每个 chunk 向量天然融入全文语境

## 语义截断问题与六种方案

问题本质:一个完整语义被切成两半后,**每一半的相关度都不够强,检索时全军覆没**——不是信息丢了,是信息被拆散后召不回来。

### 方案一:重叠切割(Overlap)——基础兜底

相邻 chunk 重叠 **chunk_size 的 10%~20%**(如 chunk_size=800、overlap=150),跨边界语义至少完整出现在一个 chunk 里。太小(5%)保护不了完整语义;太大(40%)重复内容干扰 LLM、成本上升。**只是兜底**:保证文字不丢,但解决不了"完整语义被拆散"。

### 方案二:语义边界切割

见上表——从源头避免在语义中间下刀。
![[Pasted image 20260708085338.png]]

### 方案三:句子窗口检索(Sentence Window Retrieval)

换思路:切归切,**检索后把上下文补回来**。按单句切割、单句向量化(检索粒度细、定位准);命中后把前后各 N 句一起返回给 LLM(阅读上下文完整)。代价是每句一条向量,存储量大。
![[Pasted image 20260708085357.png]]
### 方案四:父子切割(Parent-Child)

「**检索用放大镜,返回用全景图**」:同一内容存两份——小 chunk(约 200 token)建向量索引负责检索,通过 ID 关联大 chunk(约 1000 token)负责给 LLM 读。和句子窗口相比:父子是提前存好父子关系,句子窗口是命中后动态扩展;父子更可控,句子窗口实现更简单。

### 方案五:命题化切割(Propositions)

用 LLM 把文档分解成一条条**自包含的独立命题**,每条单独成立、不依赖上下文:

> 原文「企业用户享有优先客服通道,响应时间不超过 2 小时,并可申请专属技术顾问服务」
> → ①企业用户享有优先客服通道。②企业用户的客服响应时间不超过 2 小时。③企业用户可以申请专属技术顾问服务。

语义密度最高、检索精度最好,但每个 chunk 都要 LLM 调用,成本最高,适合高价值知识库。

### 方案六:Contextual Retrieval(Anthropic, 2024)

解决的根本问题:**chunk 被单独向量化时失去语境**——「此条款自 2024 年生效」单独看不知道是什么条款,向量只能编码它"看到"的文字。

做法:让 LLM **看着整篇文档**为每个 chunk 生成 1~2 句背景说明,把「背景 + chunk」拼在一起再做 Embedding 和 BM25 索引。

成本控制:配合 **Prompt Caching**,整篇文档作为公共前缀只在首次请求写入缓存(1.25 倍计费),后续 chunk 请求命中缓存按 10% 计费,文档部分成本降到 10%~20%。官方数据:100 万文档 token 的处理成本约 1 美元出头。

效果:结合 BM25 混合检索,**检索失败率(Top-20 未命中)降低约 49%**,是业界实测最显著的 chunk 质量提升方案之一。

![[Pasted image 20260708085541.png]]

### 组合建议

重叠切割做兜底 + 语义边界保证切割质量;高要求场景再加父子切割或 Contextual Retrieval。

# 四、Embedding

## 是什么:语义压缩成坐标

Embedding 模型把任意长度文本映射成**固定长度的浮点向量**(如 1024 维),本质是建立一个「语义坐标系」:**语义相近的文本,向量距离近**。

- 「苹果手机怎么截图」和「iPhone 如何截屏」一字不同,余弦相似度可达 0.95
- 「苹果手机」和「苹果汁」都有"苹果",向量距离却很远

这不是天生的,是训练出来的:模型通过**对比学习**在海量文本上"把意思相近的拉近、无关的推远"。

**为什么用余弦相似度而不是直线距离?** 高维空间里向量模长受文本长度、表达强度等非语义因素影响;余弦只看方向(夹角),把「意思相近」从「文本长短」中剥离出来。
![[Pasted image 20260708091055.png]]
## 算法三代演进

每一代都在补上一代的坑:

| 世代 | 代表 | 核心特点 | 主要局限 | RAG 适用性 |
| --- | --- | --- | --- | --- |
| 一:静态词向量 | Word2Vec、GloVe、FastText | 词级固定向量 | 处理不了多义词;词级非句子级 | 不适用 |
| 二:上下文向量 | ELMo、BERT | 同词不同语境不同向量 | 检索需两两拼接,百万库要跑百万次 | 不适用于实时检索 |
| 三:句子级对比学习 | SBERT、SimCSE、BGE、E5 | bi-encoder 独立编码,余弦可比 | 精度略低于 cross-encoder | **RAG 标配** |

第一代要点:
- Word2Vec(2013, Google):CBOW(周围词预测中心词)/ Skip-gram(中心词预测周围词);Skip-gram 样本利用率高、对低频词友好,更常用;「国王 − 男人 + 女人 ≈ 女王」
- FastText:词拆成字符 n-gram 子词,解决**未登录词**(新词只要子片段见过就能估算向量)
- 共同致命伤:「吃苹果」和「苹果手机」里的「苹果」向量完全一样

第二代要点:
- BERT 用 [CLS] 表示句子、双向注意力,语义理解强
- 但比较两句相似度必须**拼在一起**过模型:百万文档 × 几十 ms/次 = 好几小时,检索场景不可用

第三代要点:
- SBERT(2019):bi-encoder——两句**各自独立编码**,只算余弦。牺牲 token 级交叉注意的精度,换来**文档向量可提前算好存库**,查询毫秒级
- SimCSE(2021):同一句话做两次 dropout 当正样本对的对比学习;顺带解决了 BERT 句向量的**各向异性**(向量挤在窄锥形区域)问题,把语义空间"撑开"
- BGE(智源):中文 RAG 最流行开源模型,bi-encoder 和 reranker 两种形态都有

新趋势(第三代的增强,面试加分项):**指令感知 Embedding**(Qwen3-Embedding,同一文本在不同检索意图下产生不同向量)、**Matryoshka 表示学习**(前 N 维即有意义,可截断降维省存储)、**多模态 Embedding**(文本图片同空间,跨模态检索)。

## 模型选型

| 模型 | 维度 | 中文效果 | 开源 | 适用 |
| --- | --- | --- | --- | --- |
| text-embedding-3-small | 1536(可降维) | 一般 | 否(API) | 英文为主、快速上手 |
| text-embedding-3-large | 3072(可降维) | 一般 | 否(API) | 英文、精度要求高 |
| bge-large-zh-v1.5 | 1024 | 很好 | 是 | 中文知识库经典选择 |
| bge-m3 | 1024 | 好 | 是 | 中英混排、稠密/稀疏/多向量三模式 |
| Qwen3-Embedding | 多种 | 很好 | 是 | 中文新一代强力选择 |
| Voyage-3-large | 1024 | 一般 | 否(API) | 英文高精度 |

判断维度:①中英文比例(中文 BGE/Qwen3-Embedding,混排 bge-m3);②**数据合规**(不能出境就必须开源本地部署);③向量维度(精度 vs 存储/速度,百万级知识库 1024 维是平衡点)。

## 模型评估:别只看排行榜

MTEB 是通用排行榜,数据分布和你的业务(医疗/法律/客服)差异很大——**高考状元不一定擅长你这个行业的专业考试**。

正确做法:准备几百条业务「问题 + 正确 chunk」对,跑 **Hit@K** 测试(正确 chunk 是否出现在 Top-K 结果里)。**Hit@5 < 0.7 就要考虑换模型或改进 Chunking**。

# 五、向量数据库

## 是什么、为什么需要专门的库

核心能力:**近似最近邻搜索(ANN)**——给一个查询向量,在百万甚至亿级向量里快速找出最相似的 K 条。

为什么 MySQL 不行:
- B-tree 只能处理**一维有序**索引;「1024 维空间里谁离我最近」是高维综合判断,不是某一维的排序,B-tree 直接失效
- 暴力遍历是 O(N),百万条算百万次余弦,延迟不可接受
- 一句话:**MySQL 擅长精确匹配(WHERE id=123),向量数据库擅长语义相似**,不是替代关系

## 核心索引算法:HNSW vs IVF

|     | HNSW                              | IVF                  |
| --- | --------------------------------- | -------------------- |
| 结构  | 多层可导航图,逐层收窄                       | 向量聚类分桶,只搜相关桶         |
| 类比  | 找餐厅:全国→省→市→区逐层锁定                  | 图书馆分类:先找"计算机区"再细找    |
| 召回率 | 高(同延迟下 95%+)                      | 略低,需调参(nlist/nprobe) |
| 内存  | 消耗大,与向量数正比                        | 占用小                  |
| 适用  | 中大规模默认选择(Qdrant/Milvus/Chroma 默认) | 亿级超大规模               |

HNSW 关键参数(社交网络类比):
- **M**(每个节点认识几个邻居):越大图越密、精度越高、内存越大,通常 16~32
- **ef_construction**(建图时考察多少候选):通常 100~200
- **ef / search_ef**(查询时候选集大小):越大越准越慢,按需在 50~200 调

## 生产级核心能力

1. **Metadata 过滤**:先按条件(部门/时间)过滤再 ANN——如果先搜后滤,Top-K 可能大部分不满足条件,白白浪费召回名额
2. **实时增量更新**:在线写入、增量建索引,不需停服重建
3. **混合检索**:向量 + BM25 关键词一起,补精确词短板

## 选型对比

| 数据库 | 部署 | 规模 | 主要优势 | 主要劣势 |
| --- | --- | --- | --- | --- |
| Chroma | 本地/CS/云 | 中小 | 零配置上手最快,生态集成好 | 超大规模稳定性待验证 |
| Qdrant | 自托管/云,支持分布式 | 中大(亿级) | Rust 高性能,API 简洁 | 超大规模需调优 |
| Milvus | 自托管分布式 | 大(亿级) | 水平扩展,国内大厂用得多 | 部署运维复杂 |
| Pinecone | 全托管 SaaS | 中大 | 无需运维 | 费用高,数据出境合规问题 |
| pgvector | PostgreSQL 插件 | 中小 | 无新组件,可 JOIN 业务数据 | 性能弱于专用库 |

选型三维度:**数据规模、部署方式、是否需要混合检索**。常见路径:原型用 Chroma → 生产切 Qdrant → 超大规模上 Milvus;已有 PG 且量不大直接 pgvector。

## 生产实践案例(Milvus,百万级)

**规模**:150 万条 chunk × 1024 维(BGE-large-zh),HNSW(M=16, ef_construction=128)。

**内存账**:150 万 × 1024 维 × 4 字节(float32) ≈ 6GB 纯向量;进程完整跑起来 10~12GB(HNSW 图结构、metadata、管理开销)。

**实测性能**(单机 16 核 32G、ef=100):单次 top-5 查询 **P50 ≈ 20ms,P99 ≈ 60ms,100 QPS 并发稳定**。报性能数字必须带硬件和参数背景,否则没有意义。

**瓶颈一:内存不足 → swap → 延迟从 20ms 飙到 2s+**
- 解法:**标量量化 SQ8**,float32 压成 int8,内存降到 1/4(10GB→3GB),召回率损失通常 <1 个百分点,几乎无代价
- 辅助:mmap 把原始向量放磁盘、只把索引放内存

**瓶颈二:批量写入触发 Segment 合并 → 查询 P99 从 60ms 抖到 300ms+**
- 背景:新数据先进增量段,积累后合并成封存段并建索引,合并耗 CPU/IO(类似磁盘碎片整理)
- 解法:①错峰,批量写入放业务低峰(凌晨);②化整为零,每批 500~1000 条、间隔几秒,大冲击变多次小冲击

# 六、检索优化四层框架

> 检索是 RAG 的命脉:LLM 只能根据送进去的 context 回答,**检索召回的内容就是整个系统的天花板**。检索优化是投入产出比最高的环节,没有之一。

四层比喻:**索引层**决定仓库里放了什么;**查询层**决定用什么钥匙开门;**召回层**决定从哪几扇门进去找;**重排序层**决定把找到的东西里最好的几件带出来。

## 第一层:索引优化——检索粒度 vs 阅读粒度的矛盾

chunk 要同时干两件事:「检索时被找到」要求**小而聚焦**(语义不稀释);「被 LLM 读懂」要求**大而完整**(上下文不缺)。两者天然矛盾。

解法核心思路 **Small-to-Big(小块检索、大块使用)**,三种实现:

1. **Parent-Child Chunking**:子 chunk(约 150 token)建向量索引负责匹配,命中后按 parent_id 取父 chunk(约 500 token)给 LLM 读
2. **摘要索引**:LLM 为每段生成摘要,用摘要建索引(摘要语义聚焦、离问题更近),命中后返回原始段落
3. **多粒度分层索引**:同时建章节/段落/句子三层,概念性问题走章节级,细节性问题走句子级

## 第二层:查询优化——弥合提问与文档的表述鸿沟

用户口语("这东西坏了咋整")和文档书面语("产品故障排查与维修指南")在向量空间距离比直觉的远,短文本下差异被放大。四种方法:

| 方法 | 解决的核心问题 | 开销 | 适合场景 |
| --- | --- | --- | --- |
| 直接改写 | 口语化、指代不清、上下文丢失 | 1 次 LLM 调用 | 多轮对话 |
| HyDE | 问题(问句)和文档(陈述句)文体差异 | 1 次 LLM 调用 | 专业知识库 |
| Step-back 后退提问 | 问题太具体,库里只有背景原理 | 1 次 LLM 调用 | 技术文档 |
| 多 Query 扩展 | 单一提问角度覆盖不全 | N 次 LLM 调用 | 提问风格多变 |

- **直接改写**:结合对话历史补全指代,「上次说的退款的事,流程是啥」→「申请商品退款的完整操作流程是什么」
- **HyDE**(Hypothetical Document Embeddings):让 LLM 先生成一段**假设答案**,用它的向量去检索。原理:**答案到答案的距离,比问题到答案的距离近得多**;假设答案不需要准确,只需要"风格像文档",充当检索代理。注意:LLM 假设方向错了会把检索带偏,适合领域明确的知识库
- **Step-back**:「Qdrant 的 ef 参数设 100 合适吗」→ 后退成「HNSW 索引参数调优原则」,先检索背景知识再结合回答具体问题
- **多 Query 扩展**:改写成 3~5 个不同角度分别检索、合并去重,拦截网越宽捕到鱼概率越高;**原始问题必须保留在检索列表里**(改写可能丢失细节)

## 第三层:召回优化——多路召回互补盲区

### 两种检索方式的原理与盲区

| 维度 | 关键词检索(BM25) | 向量检索 |
| --- | --- | --- |
| 匹配方式 | 词汇重叠统计 | 语义空间距离 |
| 索引结构 | 倒排索引(稀疏) | 向量库(稠密) |
| 同义词 | ❌ 无法处理 | ✅ 天然支持 |
| 精确词(型号/专名/数字) | ✅ 极好 | ❌ 容易漏 |
| 可解释性 | 基于统计,可解释 | 黑盒距离 |

- **BM25** = TF(词在这篇文档出现多)× IDF(词在全库越罕见越有区分力)+ 饱和度限制(防单词重复刷分)。底层是**倒排索引**:记录"每个词出现在哪些文档"。中文需先分词(jieba)
- 经典互补案例:搜「M4 Pro 芯片参数」,向量检索可能把它和"苹果最新处理器"拉近却召不回原词条,BM25 秒中;搜「怎么退货」,文档写「申请售后」,BM25 零分,向量检索轻松命中
- **调大 top-K 救不了**:向量相似度尾部排序很平,Top-200 只是多塞无关 chunk,稀释 LLM 注意力

### 三路并行的典型组合

1. **向量检索**:管语义(同义词、表达差异)
2. **BM25**:管精确词(型号、专名、数字)
3. **多 Query 扩展**:管提问角度差异(召回覆盖率可提升 10%~20%)

### RRF 融合(Reciprocal Rank Fusion)

各路分数量纲不同(余弦 0~1 vs BM25 任意正数,像摄氏度加华氏度),没法直接加权;归一化又不稳定。RRF **只看排名不看分数**:

```
score(chunk) = Σ 1 / (k + rank)    # k 为平滑参数,通常取 60
```

多路都排名靠前的 chunk 综合分高(像多位评委都给高分的选手)。k=60 是经验平滑值,给排名靠后的候选留"保底分"。**不需训练、计算量极小**,是多路召回的标配融合方案。两路检索可并行执行,总延迟取 max 而非相加。

按场景选:有大量专有名词的知识库(电商/IT 文档)→ 向量+BM25 双路是标配;客服等提问多变场景 → 加多 Query 扩展;极高要求 → 三路全上 + Rerank。

## 第四层:重排序(Rerank)

多路召回后有 20~30 个候选,全塞给 LLM 的问题:①token 成本暴涨;②**Lost in the Middle**——LLM 处理长上下文只关注头尾,中间被忽略。需要精排到 Top-3~5。

**Bi-encoder vs Cross-encoder**:

- 向量检索是 **Bi-encoder**:query 和 chunk 各自独立编码再算余弦——像只看两个人的简历判断合不合作,快(chunk 向量可预计算)但看不到两段文字的词级关联
- Rerank 是 **Cross-encoder**:把「query + chunk」拼成一对整体输入,能看到词与词的交互——像把两个人放一个房间观察怎么配合,准得多但每个候选都要跑一次模型

所以工程上**两阶段**:粗排(Bi-encoder)从百万筛到几十条,精排(Cross-encoder)从几十条挑最好的几条。Rerank 耗时几百 ms,对体感影响小,但质量提升非常明显,是**成本效益最高的优化手段之一**。

常用模型:BGE-Reranker-v2(中英双语)、BCE-Reranker;API 有 Cohere Rerank、Jina Reranker。

## 四层组合建议

| 层次                     | 解决的核心问题          | 推荐程度             |
| ---------------------- | ---------------- | ---------------- |
| 索引优化(Parent-Child)     | 检索粒度 vs 上下文完整的矛盾 | 推荐,效果稳定          |
| 查询优化(Multi-Query/HyDE) | 提问和文档表达不对齐       | 视场景,提问质量差时必做     |
| 多路召回(向量+BM25)          | 单路检索漏召           | 推荐,低成本高收益        |
| Rerank 精排              | 粗召精度不足           | **强烈推荐**,提升精度最直接 |

典型生产级搭配:**Parent-Child 索引 + 向量/BM25 多路召回 + Rerank 精排**,提问质量差再加 Query 改写。

# 七、进阶范式与图增强

## 三代 RAG 演进

1. **Naive RAG**:提问→检索→拼 prompt→生成。两天上线 Demo,但召回什么送什么,幻觉和偏差由此而来
2. **Advanced RAG**:固定流程上加工序——检索前(Query 改写/扩展)+ 检索后(Rerank/内容压缩)。改动小、提升明显,**大多数生产系统的形态**
3. **Modular RAG**:各环节拆成可独立替换的模块(检索模块可选向量/BM25/图检索,改写模块可选 HyDE/Step-back…),像乐高按需组合。LlamaIndex Workflow、LangGraph 是这一思路的实现

朴素 RAG 的三大痛点,催生了三类高级范式:检索质量参差(→Self-RAG/CRAG)、流程死板(→Self-RAG/Agentic)、知识库有盲区(→CRAG)。

## Self-RAG:LLM 自主决策要不要检索

训练一个特殊 LLM,通过四种 reflection token 自主决定:

- Retrieval token:这个问题需不需要检索?(「1+1」直接答,省检索开销)
- Relevance token:召回的每条内容相不相关?(不相关直接跳过)
- Support token:生成的答案有没有文档支撑?(防幻觉)
- Utility token:答案质量够不够好?

流程:判断是否检索 → 逐条评估相关性 → 每个相关 chunk 生成候选答案 → 打分 → 选最优。

⚠️ 前提:这些 token **需要专门构造数据集做监督微调**,不能拿通用 GPT-4 直接套用(要么用论文开源的 Llama2-7B/13B 微调版,要么自己造数据微调)——这是它和 CRAG/Agentic RAG 最大的不同。

Self-RAG 给模型扩充一套特殊控制 Token，生成时自动输出判断标签，实现细粒度自我管控，共 4 类核心令牌.

- **[Retrieve]**：检索决策令牌
    
    - Yes：当前知识不足，触发外部检索；
    - No：内部知识足够，直接生成，跳过检索。
    
- **[IsRel]（Is Relevant）**：文档相关性判断
    
    - Relevant：该段落有用，保留；
    - Irrelevant：无关，丢弃，不进入上下文。
    
- **[IsSup]（Is Supported）**：事实支撑校验（防幻觉核心）
    
    - Supported：当前句子有检索文档证据支撑；
    - Unsupported：内容无依据，需要重新检索 / 重写。
    
- **[IsUse]（Is Useful）**：整段回答质量打分
    
    - Useful：回答完整贴合问题；
    - Not Useful：回答残缺、偏离，需补充检索。

## CRAG(Corrective RAG):检索质量差时自动纠错降级

在检索后加**质量评估器**(可用专训分类模型或 Rerank 分数近似),三级路由:

- 评估「相关」→ 正常用本地结果生成
- 评估「不相关」→ 知识库没覆盖,**丢弃本地结果、降级走网络搜索**
- 评估「模糊」→ 本地结果 + 网搜结果合并互补

核心价值是**兜底**:知识库当主力、网络搜索当备胎,覆盖不到的问题不再乱答。

## GraphRAG(微软 2024):知识图谱增强全局理解

向量检索是「一对一匹配」,只能找和问题直接相关的**局部**片段,答不了「这批文档的核心主题是什么」这类全局问题。

**预处理**:LLM 抽取实体和关系建知识图谱 → Leiden 等**社区发现算法**把紧密关联的实体聚成社区 → 每个社区生成 LLM 摘要,形成多层层次结构。

**检索**:局部问题(「A 公司 CEO 是谁」)直接图谱实体查找 + 关系遍历;全局问题用 **Map-Reduce**——LLM 分别读各社区摘要提取信息,再汇总生成答案。

适合实体关联强、需全局视角的场景(企业关系分析、药物-疾病关联、大规模文档主题归纳);代价是图构建 + 摘要的 LLM 调用成本高。

## Agentic RAG:把 RAG 做成 Agent 循环

前面的范式流程仍是预定义的;有些问题连「检索几次、每次检索什么」都无法预先定义。Agentic RAG 把 RAG 嵌入 Agent 循环,LLM 每轮自主决定:信息够了吗?不够下一步搜什么关键词?够了就生成。设**最大迭代次数**防死循环。

适合多步骤推理(「分析 A 公司近三年财报,找出营收放缓根因」)。**流程不是写死的,而是运行时 LLM 自己决定**——Self-RAG 是加了"要不要检索"的开关,CRAG 是加了"检索不好怎么办"的兜底,Agentic RAG 是把整个流程变成可循环调整的智能体。

## 范式对比

| 范式           | 核心创新              | 适合场景          | 工程复杂度       |
| ------------ | ----------------- | ------------- | ----------- |
| Advanced RAG | Query 改写 + Rerank | 大多数生产场景       | 低           |
| Self-RAG     | LLM 自主决策检索        | 问题类型多样、部分无需检索 | 高(需特殊训练)    |
| CRAG         | 质量差时降级网搜          | 知识库覆盖不全       | 中           |
| GraphRAG     | 社区发现 + 层次摘要       | 关系复杂、需全局理解    | 高(图构建成本大)   |
| Agentic RAG  | 多轮动态检索            | 复杂多步骤问题       | 中(Agent 框架) |

## 什么时候用图数据库增强

向量检索的根本局限:**单跳**——每次只能找和问题直接相关的内容,没有"关系"概念,不知道实体 A 和 B 之间有边、该沿哪条边走。"多检索几次"救不了,因为你不知道下一步该搜什么。

向量检索答不了的问题(答案藏在多节点路径上,没有一篇文档直接写出来):
- 「小米的主要竞争对手的 CEO 是谁」(实体跳转,第二跳就断)
- 「治 A 病的药和治 B 病的药有没有相互作用」(药物→靶点→相互作用)
- 「这个函数直接和间接依赖的库有哪些漏洞」(依赖链逐层展开)

**组合用法**:向量检索做**入口**(定位问题相关实体)→ 图数据库接力做**关系遍历**(沿边一跳跳收集关联信息)→ 两路上下文合并给 LLM。类比:向量检索是导航定位(在哪),图遍历是沿路线一站站走(能到哪)。

**判断原则**:问题里是否同时出现多个具体实体、并且在问它们之间的关系?是 → 考虑图增强;只是找描述性内容 → 向量 + Rerank 就够,强上图是过度设计(实体/关系抽取成本高、易错、维护复杂)。

# 八、幻觉防控

## 两类幻觉,根源不同

RAG ≠ 无幻觉。塞进 prompt 不等于 LLM 一定"老老实实照着说":

1. **检索层幻觉:没找到,LLM 靠猜**。检索失败时 prompt 里空的或全不相关,LLM 不会说"不知道",会用训练时的知识编一个——知识库写「7 天无理由退款」,检索没召回,LLM 答「30 天」,用户照做失败
2. **生成层幻觉:找到了,但 LLM 超发挥**。更隐蔽更常见:以资料为基础加了自己的推断、补充了原文没有的细节、把不相关的信息拼在一起

## 四个方案,从低成本到严格递进

### 方案一:Prompt 强约束(治生成层)

```
回答规则:
- 只能使用【参考资料】中的信息来回答,不得引入资料之外的知识
- 如果参考资料中没有足够信息,必须明确回答:「根据现有资料,无法回答该问题」
- 回答时标注信息来源(来自哪条参考资料)
- 不要推断、不要猜测、不要补充资料中没有明确说明的内容
```

设计意图:第一条让 LLM 知道边界(传话人,不是百科全书);第二条给它**合法的逃生出口**(允许优雅认怂);第三条强制自检对照原文。

### 方案二:检索质量门控(治检索层)

Rerank 给候选打 0~1 相关度分,**最高分低于阈值直接拒答**(「知识库无相关信息,建议联系人工」),不让 LLM 在低质量上下文上硬撑。

- 阈值无通用值,拿「答案在库内/不在库内」两类测试集跑分数分布找切点,经验区间 **0.3~0.6**(金融/医疗偏高,闲聊偏低)
- 核心认知:**答错比不答更危险**,拒答不是笨,是负责

### 方案三:生成后引用核查(事后审稿)

答案生成后,再发起一次 LLM 调用当"编辑":逐条核查答案里的每个关键声明在 chunk 里有没有依据,没有的标「无法核实」或删掉。代价是延迟和成本翻倍,只用于医疗/法律等极高准确性场景。

### 方案四:结构化输出强制溯源

让 LLM 输出 JSON,每条结论必须带来源编号:

```json
{
  "answer": "完整回答",
  "statements": [
    {"claim": "具体结论1", "source_ids": [1, 2]},
    {"claim": "具体结论2", "source_ids": [3]}
  ],
  "confidence": "high/medium/low"
}
```

有效原因:①LLM 构建 JSON 时必须主动想"这条结论从哪来"(像强制标注参考文献);②系统可**程序化验证** source_ids 与 claim 是否匹配,不匹配自动过滤。

## 组合策略

- 普通企业知识库/客服:**方案一 + 方案二**,上线前必做,成本极低,幻觉解决大半
- 高可信要求(金融/法律):再加方案四
- 极高要求(医疗/合规):四个全上,用延迟和成本换准确性

> 核心原则:**治幻觉,先治检索**。检索到了正确内容 + Prompt 稍加约束,幻觉就少了大半;检索这步烂,再多生成层约束也填不了坑。

# 九、效果评估

## 为什么必须系统评估

靠"用户投诉"是亡羊补牢,靠"人工抽查几十条"样本太少且主观。没有指标体系 = 盲飞:改了 Chunking/换了模型,不知道变好还是变坏;出了问题不知道是检索的锅还是生成的锅;优化了无法验证收益。

评估难点:答案是自然语言、无唯一标准答案;分不清哪层出问题;人工标注成本高。**解法:分层评估。**

## 检索层指标

需准备「问题 + 正确 chunk ID」测评集:

- **Hit@K**:「找到没」——Top-K 结果里有没有正确 chunk。Hit@5 < 0.7 → 检索层有问题(换 Embedding / 改 Chunking);> 0.8 → 检索 OK,问题在生成层
- **MRR**(平均倒数排名):「多快找到」——对每个问题算 1/排名再平均(第 1 名 1 分、第 2 名 0.5、第 5 名 0.2)。MRR < 0.5 → 内容召回了但排不到前面,Rerank 效果差

公式(N = 测评问题总数):

$$\text{Hit@}K=\frac{1}{N}\sum_{i=1}^{N}\mathbb{1}[\text{正确 chunk} \in \text{Top-}K_i],\qquad \text{MRR}=\frac{1}{N}\sum_{i=1}^{N}\frac{1}{\text{rank}_i}$$

$\mathbb{1}[\cdot]$ 是指示函数(命中记 1、否则 0);$\text{rank}_i$ 为第 i 个问题**第一个**正确 chunk 的排名(没命中则 $1/\text{rank}_i=0$)。所以 Hit@K 是「命中率」,MRR 是「排名倒数的均值」。

两者配合:Hit@5=0.9 但 MRR=0.3,说明找得到但排得靠后。

## 生成层指标:RAGAs 框架

核心思路 **LLM-as-a-Judge**(LLM 当裁判自动打分,免人工标注)。四个指标:

| 指标                       | 问的问题                      | 目标值   |
| ------------------------ | ------------------------- | ----- |
| Faithfulness 忠实度         | 答案每句话在 chunk 里有出处吗?(衡量幻觉) | > 0.8 |
| Answer Relevancy 答案相关性   | 答的是用户问的那个问题吗?(防跑题)        | > 0.8 |
| Context Recall 上下文召回率    | 回答所需信息被检索结果覆盖了多少?(需标准答案)  | > 0.7 |
| Context Precision 上下文精确率 | 检索结果里相关内容是否排在前面?(噪音多不多)   | —     |

这四个指标不是「算一次余弦相似度」那么简单,核心算法各不相同:

**① Faithfulness 忠实度——拆句 + 逐句核验**

1. LLM 把答案拆成若干条独立**陈述**(claim)
2. 对每条陈述,让 LLM 判断能否由检索 context 推出(支持 / 不支持)
3. 支持比例即为得分:

$$\text{Faithfulness}=\frac{\#\,\{\text{能被 context 支持的陈述}\}}{\#\,\{\text{答案拆出的全部陈述}\}}$$

例:答案拆出 5 条陈述,4 条在 context 有据、1 条编造 → 0.8。

**② Answer Relevancy 答案相关性——反推问题比相似度**

1. LLM 基于答案**反向生成** n 个问题(通常 n=3~5)
2. 每个反推问题与原问题各算一次 embedding 余弦相似度
3. 取平均:

$$\text{Answer Relevancy}=\frac{1}{n}\sum_{i=1}^{n}\cos\!\big(E_{q_i},\,E_{q}\big)$$

$E_{q_i}$ 是第 i 个反推问题的向量,$E_{q}$ 是原问题向量。直觉:答案切题,从它反推出的问题就该贴近原问题。答「我不知道」这类会被判定为 noncommittal 直接记 0。

**③ Context Recall 上下文召回率——需标准答案,逐句归因**

1. 把标准答案(ground truth)拆成句子
2. 逐句判断该句信息能否在检索 context 里找到归属
3. 可归因比例即为得分:

$$\text{Context Recall}=\frac{\#\,\{\text{能归因到 context 的标准答案句}\}}{\#\,\{\text{标准答案全部句子}\}}$$

**④ Context Precision 上下文精确率——相关项是否排在前面**

对 Top-K 逐位算 Precision@k,再按该位是否相关加权平均(等价于平均精度 AP):

$$\text{Context Precision@}K=\frac{\sum_{k=1}^{K}\big(\text{Precision@}k\times v_k\big)}{\#\,\{\text{Top-K 中的相关项}\}},\qquad \text{Precision@}k=\frac{\#\,\{\text{前 }k\text{ 项中的相关项}\}}{k}$$

$v_k\in\{0,1\}$ 表示第 k 位是否相关。相关 chunk 越靠前,分越高。

> 💡 ①③ 的本质是「拆句 + LLM 逐句裁判」,每条陈述 / 每句都要调一次 LLM——这正是 RAGAs 贵、必须抽样的根本原因。

⚠️ 易混淆:Faithfulness 问「说的是不是真的」,Answer Relevancy 问「说的是不是用户想要的」——问北京天气,答了一篇全对的北京历史:忠实度高、相关性低。

成本控制:RAGAs 每次评估都调 LLM,几千条全量跑很贵。缓解:①抽样最有代表性的 200~500 条;②评判模型降级(如 GPT-4o → GPT-4o-mini,成本降 10 倍)。

## 用指标定位问题

| 指标低 | 说明 | 优化方向 |
| --- | --- | --- |
| Hit@K | 正确 chunk 没被召回 | 换 Embedding、调 Chunking |
| MRR | 召回了但排名靠后 | 加强 Rerank |
| Context Recall | 检索缺少答题必要信息 | 更强 Embedding、多路召回 |
| Context Precision | 召回太多噪音稀释注意力 | 加强 Rerank、减少送 LLM 的 chunk 数 |
| Faithfulness | LLM 在编造 | Prompt 强约束、引用核查、质量门控 |
| Answer Relevancy | 答案跑题 | 改 Prompt 指令(严格回答问题本身) |

## 线上指标:最终验收标准

- **踩率**:最直接的负反馈
- **追问率**:用户重复问同一问题 → 答非所问
- **转人工率**:太高说明覆盖不足;但因加质量门控而升高不一定是坏事(宁转人工不给错误答案)
- **空回答率**:「我不知道」比例过高 → 知识库需扩充
- **会话解决率**:一次对话能否解决问题,最贴近用户体验的综合指标

**评估闭环**:离线测评(Hit@K + RAGAs,快速迭代定位)→ 上线 → 线上观测(踩率/转人工率,最终验收)→ 发现问题 → 离线复现 → 修复 → 再上线。离线好不代表线上好(测试集不能代表真实分布;Faithfulness 优化过头 Prompt 收太死,回答保守用户反而不满),两者要定期交叉对照。

# 十、知识库动态更新

> 动态更新是 RAG 投产的必备条件:产品手册改版、政策更新,知识库不跟进就一直返回过期信息。

## 为什么比更新普通数据库麻烦

普通数据库一条记录直接 UPDATE;RAG 里**文档和向量库是一对多**(一篇文档 → 几十上百个 chunk)。文档内容一变,**切割边界就变了**——原来第 3 个 chunk 的内容可能散在新的第 3、4 个 chunk 里,旧新 chunk 根本无法一一对应做"局部打补丁"。

> 类比:装修拆了一堵墙重建,不能指望原来的插座位置还对得上,整面墙电路要重新布。

**唯一可靠的做法:先删后增**——把该文档之前入库的所有 chunk 全删掉,按新内容重新切割入库。暴力但不出 bug。

三种变更操作:新增(直接走完整入库流程)、修改(先删后增)、删除(清掉所有对应 chunk,不留"僵尸 chunk")。

## 怎么检测文档变没变:内容 hash

入库时算文档内容的 MD5/SHA256,和文档 ID、chunk ID 列表一起存(Redis/数据库均可)。下次对比 hash:相同跳过,不同触发重处理。改一个标点 hash 就完全不同,不会漏;计算成本极低。

进一步优化:先用「最后修改时间」粗筛,只对时间戳变了的文档算 hash——百万篇文档真正改过的可能只有几千篇,过滤掉 99%。

## chunk ID 设计(前期不设计,后期很狼狈)

删除一篇文档的所有 chunk,前提是能快速找到"这篇文档对应哪些 chunk"。两种做法等效:

- chunk ID 带文档前缀:`product_manual_v3_chunk_001`,按前缀批量删
- metadata 里存 `source_doc_id`,按 metadata 过滤批量删

## 变更感知方式对比

| 方案 | 延迟 | 复杂度 | 适用 |
| --- | --- | --- | --- |
| 定时轮询 | 分钟~小时级 | 低 | 更新频率低、实时性要求不高(内部知识库) |
| Webhook 触发 | 秒级 | 中 | 数据源支持(Confluence/Notion/语雀 保存即推送) |
| 消息队列(Kafka 等) | 秒级 | 中高 | 大规模高并发更新,生产首选 |
| 全量重建 | 分钟~小时级 | 低 | 库很小,或换了 Embedding 模型/Chunking 策略导致新旧向量不兼容 |

## 灰度更新(蓝绿部署思路)

核心生产库直接删旧写新风险大。稳妥做法:新 chunk 打 `version=new` 标签并行写入、旧版保留 `version=old` → 用测试问题集同时跑新旧版本对比质量 → 验证通过后把检索过滤条件从 old 切到 new → 最后清理旧版。**切换秒级、出问题立刻回滚**,金融/医疗等高要求场景必备。

**生产推荐组合:事件驱动 + hash 变更检测 + 先删后增。**

# 十一、落地难点与核心心法

> Demo 一两天能跑起来,调到生产可用往往要几周到几个月。每个环节都可能是瓶颈且相互影响。

## 三大难点

**① 文档预处理(地基型瓶颈,最脏最累)**

- pypdf 这类通用库做文本流提取,遇到表格/双栏/嵌套排版就乱:产品规格表解析成「型号内存价格iPhone 158GB5999」,行列关系全丢——**进去的是垃圾,出来的也是垃圾**,Embedding 再好也救不回
- 解法:pdfplumber 专攻表格、unstructured 按格式专项处理;高价值文档(合同/财报/专利)可用多模态 Vision 模型理解 PDF 截图,但成本高几十到上百倍,只适合单价高、数量可控的文档
- 其他坑:扫描件要 OCR、图片内文字提取不到、代码块切割破坏逻辑。生产系统的预处理代码量往往比 RAG 核心逻辑还多

**② 检索质量调优**(问题来源多、排查费劲):Chunking 切散内容、Query 语义鸿沟、精确词召回差——三个问题交织,系统解法见第六章四层框架。

**③ 效果评估**(不量化就是瞎猜):分层指标体系见第九章。

## 核心心法(一句话记住每个主题)

| 主题 | 心法 |
| --- | --- |
| 定位 | RAG = 给 LLM 开卷考试;微调管「怎么说」,RAG 管「说什么」 |
| 上限 | 检索质量是整个系统的天花板,LLM 只是在复述检索结果 |
| 存储 | 向量负责找到,原文负责阅读;一篇文档 = 多条记录 |
| 切割 | 500~1000 token 起步,overlap 10%~20% 兜底;小块检索、大块阅读 |
| Embedding | 必须同一个模型建库和查询;选型看业务数据 Hit@K,不迷信排行榜 |
| 检索 | 向量管语义、BM25 管精确词,RRF 融合,Rerank 收尾 |
| 幻觉 | 治幻觉,先治检索;答错比不答更危险,给 LLM 合法的「不知道」出口 |
| 评估 | 分层量化:检索层 Hit@K/MRR,生成层 RAGAs,线上指标做最终验收 |
| 更新 | 先删后增,永远不做 chunk 局部更新;hash 检测变更 |
| 落地 | 原型两天,生产数月;文档预处理是地基,烂进烂出 |
