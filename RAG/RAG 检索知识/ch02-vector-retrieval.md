### 【必须掌握】Embedding 是什么

Embedding 模型的目标：把任意文本映射到一个固定维度的实数向量，使得**语义相近的文本，向量方向相近**。

这个"方向相近"用**余弦相似度**度量：

$$\cos(\theta) = \frac{\vec{A} \cdot \vec{B}}{|\vec{A}| \cdot |\vec{B}|} = \frac{\sum_{i=1}^{n} A_i B_i}{\sqrt{\sum_{i=1}^{n} A_i^2} \cdot \sqrt{\sum_{i=1}^{n} B_i^2}}$$

余弦相似度的范围是 [-1, 1]：
- 1：方向完全相同（语义相同）
- 0：方向垂直（语义无关）
- -1：方向相反（语义相反，NLP 中很少出现）

**为什么用余弦而不用欧式距离？**

欧式距离的公式是两点之间的直线距离：

$$d_{euclidean}(\vec{A}, \vec{B}) = \sqrt{\sum_{i=1}^{n}(A_i - B_i)^2}$$

它同时受**方向**和**模长（向量的长度）**影响。用一个具体例子说明问题：

```
向量 A = [1, 0]   → 模长 = 1
向量 B = [3, 0]   → 模长 = 3

两者方向完全相同（语义一致），但欧式距离 = 2，不为 0
余弦相似度 = 1（正确反映语义相同）
```

在文本场景里，长文档的 Embedding 向量模长天然比短查询大，导致欧式距离虚高——即便语义完全匹配，距离也会因为"长度差异"被拉大。

**余弦相似度只看方向，忽略模长：**

$$\cos(\theta) = \frac{\vec{A} \cdot \vec{B}}{|\vec{A}| \cdot |\vec{B}|}$$

分母把两个向量的模长都除掉了，剩下的只是方向的夹角，所以不受长度影响。

---

**L2 归一化之后为什么两者等价？**

L2 归一化就是把向量除以自身的模长，让所有向量的模长都变成 1：

$$\hat{v} = \frac{\vec{v}}{|\vec{v}|}$$

归一化后，展开欧式距离的平方：

$$d^2 = |\hat{A} - \hat{B}|^2 = |\hat{A}|^2 + |\hat{B}|^2 - 2(\hat{A} \cdot \hat{B}) = 1 + 1 - 2\cos(\theta) = 2(1 - \cos\theta)$$

所以：

$$d_{euclidean} = \sqrt{2(1 - \cos\theta)}$$

$\cos\theta$ 越大（语义越近），欧式距离越小——**两者单调等价，排名结果完全一致**。

上面那个例子，归一化后：

```
向量 A = [1, 0]  归一化 → [1, 0]
向量 B = [3, 0]  归一化 → [1, 0]

欧式距离 = 0  ✅  余弦相似度 = 1  ✅  结果一致
```

现代 Embedding 模型（包括 bge、OpenAI text-embedding 系列）输出时都已经做了 L2 归一化，所以 Milvus 用欧式距离还是余弦距离在这类模型下效果是一样的。

### 现代 Embedding 模型架构

主流方案是基于 **Transformer** 的双编码器（Bi-Encoder）：

```
文本输入
  → Tokenizer（分词）
  → Transformer（多层自注意力）
  → [CLS] token 向量 / 平均池化
  → L2 归一化
  → 固定维度向量（如 768 维、1536 维）
```

关键设计决策：

| 决策 | 选项 | 影响 |
|------|------|------|
| 向量维度 | 384 / 768 / 1536 | 维度越高精度越好，存储和计算越贵 |
| 池化方式 | CLS token / Mean Pooling | Mean Pooling 对长文本通常更稳定（见下方分析） |
| 训练目标 | 对比学习（InfoNCE Loss） | 正样本对拉近，负样本对推远（见下方推导） |
| 最大 token 长度 | 512 / 8192 | 超长文本被截断，chunk 策略很重要 |

#### 【理解即可】InfoNCE Loss 推导：对比学习如何训练 Embedding

> **为什么要理解这个推导？** 它解释了两个工程中直接有用的结论：①**batch 越大效果越好**（负例越多，区分度越细）；②**温度 τ 推荐 0.05 而非 0.5**（锐利分布才能有效区分正负例）。微调 Embedding 模型或选择训练参数时，这两点是关键决策依据。

InfoNCE（Information Noise Contrastive Estimation，Oord et al. 2018）是现代 Embedding 模型的标准训练目标，理解它有助于理解"为什么负例数量重要"和"温度系数是什么"。

**训练设置：**

一个 batch 包含 $N$ 对正例 (查询, 相关文档)：$(q_1, d_1^+), (q_2, d_2^+), ..., (q_N, d_N^+)$

对每个查询 $q_i$，**batch 内其他文档** $\{d_j^+ | j \neq i\}$ 被当作**批内负例（in-batch negatives）**——总计 $N-1$ 个负例。

**InfoNCE 损失：**

对查询 $q_i$，在 $N$ 个候选文档中识别出正例 $d_i^+$ 的概率：

$$P(d_i^+ | q_i) = \frac{\exp(\text{sim}(q_i, d_i^+) / \tau)}{\sum_{j=1}^{N} \exp(\text{sim}(q_i, d_j^+) / \tau)}$$

InfoNCE 损失让这个概率最大（即最小化负对数概率）：

$$\mathcal{L}_i = -\log P(d_i^+ | q_i) = -\log \frac{\exp(\text{sim}(q_i, d_i^+) / \tau)}{\sum_{j=1}^{N} \exp(\text{sim}(q_i, d_j^+) / \tau)}$$

整个 batch 的损失取平均：

$$\mathcal{L} = \frac{1}{N} \sum_{i=1}^{N} \mathcal{L}_i$$

其中 $\tau$（tau）是**温度超参数**，通常取 0.05-0.1。

**温度 τ 的效果：**

| $\tau$ | 概率分布 | 训练效果 |
|--------|---------|---------|
| 0.01 | 极锐（接近 one-hot） | 极度惩罚"几乎正确但不是最好"的向量，收敛难 |
| **0.05** | **锐（推荐值）** | **显著区分正负例，训练稳定** |
| 0.5 | 平缓 | 对负例容忍度高，正负例区分不明显，效果差 |

**梯度分析——损失函数如何"推动"向量方向：**

对正例相似度的梯度（方向：增大）：

$$\frac{\partial \mathcal{L}_i}{\partial \text{sim}(q_i, d_i^+)} = \frac{1}{\tau}(P(d_i^+ | q_i) - 1) < 0$$

负数梯度意味着：通过梯度下降更新后，$\text{sim}(q_i, d_i^+)$ 会**增大**（正例被拉近）。

对负例相似度的梯度（方向：减小）：

$$\frac{\partial \mathcal{L}_i}{\partial \text{sim}(q_i, d_j^+)} = \frac{1}{\tau} P(d_j^+ | q_i) > 0$$

正数梯度 → $\text{sim}(q_i, d_j^+)$ 会**减小**（负例被推远）。

**两个效果叠加：** 正例被拉向 $q_i$，负例被推离 $q_i$——这正是"让语义相近的文本向量方向相近"的直接实现。

**为什么 batch 越大效果越好：**

$N$ 越大，每个查询有越多的负例 → 更细致地区分"哪些文档不是正例"。BGE 模型的训练 batch 通常设为 4096-8192，而不是普通 32-64。

**信息论视角（"InfoNCE"中"Info"的含义）：**

InfoNCE 损失是查询和文档之间**互信息（Mutual Information）**的下界：

$$I(q; d) \geq \log N - \mathcal{L}$$

最小化 $\mathcal{L}$ ≈ 最大化 $I(q; d)$——让查询向量包含尽可能多的"能识别对应文档"的信息。

#### [CLS] vs Mean Pooling：理论上的区别

Transformer 最后一层输出 $n$ 个 token 的隐向量，需要聚合成一个固定向量。有两种主要方案：

**[CLS] Token 池化：**

直接取第一个 [CLS] token 的隐向量作为句子表示：

$$\vec{v}_{\text{CLS}} = h_{\text{[CLS]}}$$

预训练时，BERT 用 [CLS] 做 NSP（Next Sentence Prediction）任务，期望它能编码"整句摘要"。

**Mean Pooling（均值池化）：**

对所有（非特殊）token 的隐向量取均值：

$$\vec{v}_{\text{mean}} = \frac{1}{|S|} \sum_{t \in S} h_t$$

其中 $S$ 是排除 [CLS]、[SEP]、[PAD] 的真实 token 集合。

**为什么 Mean Pooling 在长文本上通常更好：**

**1. 信息压缩瓶颈：**

[CLS] 是单个向量，所有 token 的信息必须通过注意力机制"汇聚"到这一个位置。对于短文本（<50 tokens），这个压缩基本无损；对于长文本（200-500 tokens），部分 token 的信息可能无法充分流向 [CLS]。

Mean Pooling 不存在这个瓶颈——每个 token 的贡献是显式的、平等的。

**2. 表示退化问题（Representation Degeneration）：**

Gao et al.（2019）发现：直接用预训练 BERT 的 [CLS] 向量做句子相似度，效果差的原因之一是所有句子的向量都集中在**高频词主导的一个狭窄锥形区域**，向量之间的余弦相似度虚高（即使不相关的句子也有 0.9+ 的相似度）。

Mean Pooling 相对缓解这个问题，因为均值对词频偏置有一定平滑效果。

**3. SimCSE 的实验结论：**

SimCSE（Gao et al. 2021）在 STS 基准上系统对比了 [CLS] 和 Mean Pooling（在相同的对比学习框架下），均值池化在大多数数据集上表现略优或持平，无一案例显著劣于 [CLS]。

**何时 [CLS] 是合理选择：**

- 模型专门为 [CLS] 微调（如 DPR、ColBERT-v1）
- 推理效率优先（只需存一个 token 的向量，省内存）
- 文本平均长度 < 64 tokens（短文本压缩损失可忽略）

**实践总结：**

优先选择输出说明中声明"Mean Pooling"的模型（如 BGE、GTE 系列）。使用 OpenAI text-embedding API 时，内部已处理好（无需选择），直接使用 API 输出即可。

### ANN 索引原理

精确最近邻（KNN）需要计算查询向量与所有文档向量的相似度——百万量级时需要几秒，不可接受。

**HNSW（Hierarchical Navigable Small World）**——Milvus 的默认索引：

#### 从"小世界"现象说起

1967 年社会学家 Milgram 做了一个实验：让美国任意两个陌生人通过"朋友的朋友"传信，平均只需要 **6 步**就能连通——这就是"六度分隔"理论。

背后的原因：社交网络里大多数连接是**局部的**（朋友住在附近），但存在少数**长程连接**（认识远方的人），正是这些长程连接让整个网络的"直径"变得很小。

HNSW 把这个思想搬进了向量检索：构建一个图，节点是向量，边是相邻关系。图里既有短程的精细连接，也有长程的跳跃连接，就能用很少的跳数快速找到最近邻。

#### 为什么要"分层（Hierarchical）"

单层图已经能做近似搜索，但有个问题：**起点选在哪里？** 如果起点选得不好，搜索要走很多步才能靠近目标。

HNSW 的解法是把图分成多层，越高层节点越少、连接越稀疏，专门用来做"粗定位"：

```
层 3（最稀疏）：  A ←——————————→ Z       只有少数节点，长程跳转，快速缩小范围
层 2：          A → C ————→ X → Z       中等密度，中程跳转
层 1：          A → B → C ... → Z       较密，短程精细
层 0（最密集）：  所有节点都在，局部邻居图   最终精确定位
```

每个节点随机决定自己出现在哪几层——用指数分布采样，所以大多数节点只在底层，少数节点出现在高层（类似跳表）。

#### 查询过程（从上往下，逐层收敛）

```
1. 从最高层的固定入口节点出发
2. 在当前层贪心搜索：找当前节点的所有邻居，走向距离目标最近的那个
3. 到达当前层局部最优（没有更近的邻居）后，下沉到下一层
4. 重复直到第 0 层，在底层维护一个候选集，返回 top-k
```

关键在于：高层的长程跳跃帮你快速"跑到"目标附近，底层的密集连接帮你精确找到最近邻。整体跳数是 $O(\log N)$，而暴力 KNN 是 $O(N)$。

#### 插入新节点的过程

```
1. 随机采样决定该节点进入哪几层（如采样到层 2，则出现在层 0、1、2）
2. 从最高层开始，用查询过程找到每层的最近邻
3. 在每层建立双向边（该节点 ↔ 它的邻居）
4. 如果某个节点的边数超过 M，剪掉最远的边
```

**HNSW 关键参数：**

| 参数               | 默认值 | 作用             | 调大的代价          |
| ---------------- | --- | -------------- | -------------- |
| `M`              | 16  | 每个节点在每层的最大连接数  | 内存增大，索引变慢，精度提升 |
| `efConstruction` | 200 | 建索引时每层维护的候选集大小 | 建索引更慢，索引质量更高   |
| `ef`（查询时）        | 50  | 查询时底层维护的候选集大小  | 查询更慢，召回率提升     |

经验值：`ef` 至少设为 `top_k` 的 4~8 倍，否则高精度查询会漏结果。

**IVF（Inverted File Index）**——大规模场景替代方案：

1. 用 K-Means 对所有向量聚类（如 1000 个簇）
2. 查询时先找最近的几个簇（`nprobe`），只在这些簇内搜索
3. 大幅减少计算量，代价是可能漏掉跨簇的近邻

### 常见踩坑

**坑1：入库和查询必须用同一个模型**

向量空间由模型定义，不同模型不兼容。换模型后必须重新入库所有文档。

**坑2：模型的 max_token 限制**

绝大多数 Embedding 模型最大输入是 512 tokens（约 350 中文字）。超出被截断，后半部分语义丢失：

```python
# 错误：不检查长度直接编码
embedding = model.encode(very_long_text)  # 超出 512 会被截断

# 正确：确保 chunk 长度在模型限制内
MAX_TOKENS = 512
if len(tokenizer.encode(text)) > MAX_TOKENS:
    text = tokenizer.decode(tokenizer.encode(text)[:MAX_TOKENS])
```

**坑3：Milvus 的 ef 参数影响查询精度**

默认 ef=10 在大规模数据下精度会下降：

```python
search_params = {
    "metric_type": "COSINE",
    "params": {
        "ef": 64  # 至少是 top_k 的 4~8 倍
    }
}
```

向量模型在训练时用的是通用语料，但专业领域文档（法律、财务、工程规范）的向量质量有保证吗？

---

**🔬 研究员**

这是向量检索最核心的挑战之一——**领域分布偏移（Domain Shift）**。

通用模型在专业领域表现下降的原因：
1. 专业术语训练数据少，向量不准确
2. 同一词在不同领域含义不同（"期限"在法律和金融中不同）
3. 专业文体与通用文本差异大

解决方案按成本排序：

| 方案 | 效果 | 成本 |
|------|------|------|
| 用领域专用模型 | 好 | 低（直接用） |
| 在领域数据上继续预训练 | 很好 | 高（需要大量计算） |
| 对比学习 Fine-tune | 非常好 | 中（需要标注数据） |
| 使用 BGE-M3（多语言多粒度） | 较好 | 低（本地免费） |
