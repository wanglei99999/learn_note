# 第十二章：RAG vs Fine-tuning——什么时候用哪个
### —— 三方深度对话：研究员 × 工程师 × 提问者

> 原始来源：02-advanced-retrieval.md，第十二章（第609-721行）。本章讲解 RAG 与 Fine-tuning 的本质区别、选择框架、最强组合（RAG + Fine-tuning），以及生产环境 RAG 的延迟分解与 500ms SLA 优化手段。

---

**🔬 研究员**

两者不是非此即彼，而是解决不同问题：

### 本质区别

| | RAG | Fine-tuning |
|--|-----|-------------|
| **知识存储位置** | 外部向量库（运行时检索） | 模型参数（训练时注入） |
| **知识更新** | 只需更新向量库，无需重新训练 | 需要重新训练或微调 |
| **知识边界** | 清晰（只有库里的文档） | 模糊（混合了预训练知识） |
| **可解释性** | 有引用来源 | 无法解释知识从哪来 |
| **成本** | 检索 + 推理（持续） | 训练（一次性）+ 推理 |

### 选择框架

```
                    知识时效性
                    ┌──────────────────────────────────┐
                    │  频繁变化        静态/缓慢变化     │
                    │                                  │
 需要引用来源  ✅   │    RAG           RAG              │
                    │                                  │
 不需要引用    ❌   │    RAG           Fine-tune 或      │
                    │                 RAG + Fine-tune  │
                    └──────────────────────────────────┘
```

**用 RAG 的场景：**
- 知识需要实时更新（新闻、政策、价格）
- 需要引用来源，结果可溯源
- 知识库是私有数据，不能训练到模型里
- 多租户场景，不同用户有不同知识库

**用 Fine-tuning 的场景：**
- 需要改变模型的**行为风格**（语气、格式、专业度）
- 特定领域的**推理模式**（法律推断逻辑、医学诊断思路）
- 知识库非常稳定，不频繁更新
- 对响应延迟要求极高（无需检索步骤）

**最强组合：RAG + Fine-tuning**

```
Fine-tuning：让模型学会"如何理解这个领域的语言和推理方式"
RAG：       让模型获取最新的、具体的知识内容

例：法律问答系统
  Fine-tune：训练模型理解法律文书格式、推理方式
  RAG：     检索最新法规条文、判例
```

**🔧 工程师**

### Fine-tuning 的实际门槛

**数据要求：**

| Fine-tuning 类型 | 数据量 | 成本 |
|----------------|--------|------|
| LoRA（参数高效微调） | 1000~5000 条 | 相对低 |
| 全参数 Fine-tune | 10000+ 条 | 高 |
| 数据标注本身 | — | 每条 $0.5~5 |

**效果容易被高估的原因：**

Fine-tune 后模型在测试集上表现更好——但这可能只是因为训练数据和测试数据来自同一分布。遇到训练集中没有的新问题，Fine-tune 的模型不一定比原始模型好。

**实践建议：先做 RAG，再考虑 Fine-tuning**

很多"需要 Fine-tuning"的问题，其实是 Prompt Engineering 没做好或 RAG 质量差造成的。先优化 RAG + Prompt，再评估是否需要 Fine-tuning。

**🔬 研究员**

### LoRA 的数学原理：为什么低秩就够了

LoRA（Low-Rank Adaptation，Hu et al. 2021）是目前最主流的参数高效微调方法。理解它的数学有助于选对超参数，也能解释它的边界在哪里。

#### 核心公式

对预训练模型中一个权重矩阵 $W_0 \in \mathbb{R}^{d \times k}$（如注意力层的 $W_Q$），LoRA 不直接修改 $W_0$，而是注入一个**低秩增量**：

$$W' = W_0 + \Delta W = W_0 + BA$$

其中：
- $W_0$：**冻结**（frozen），反向传播不更新
- $B \in \mathbb{R}^{d \times r}$：**可训练**，初始化为全零
- $A \in \mathbb{R}^{r \times k}$：**可训练**，初始化为高斯随机
- $r \ll \min(d, k)$：秩（rank），控制可训练参数数量

前向传播：

$$h = W'x = W_0 x + BAx = \underbrace{W_0 x}_{\text{冻结，不反传}} + \underbrace{BAx}_{\text{可训练部分}}$$

#### 初始化策略的设计意图

$B$ 初始化为 0，$A$ 初始化为随机高斯——这个组合确保：

$$\Delta W = BA = \underbrace{0}_B \cdot A = 0 \quad \text{（训练开始时）}$$

训练开始时 LoRA 不改变模型行为，完全等同于原始基础模型。这保证了**训练稳定性**：从一个已知的好起点（预训练模型）开始优化，而不是从随机点出发。

实践中还会加一个缩放系数 $\frac{\alpha}{r}$：

$$h = W_0 x + \frac{\alpha}{r} BAx$$

$\alpha$ 是超参数，调 $\alpha/r$ 的比值相当于调学习率。$\alpha = r$ 时退化为标准 LoRA，$\alpha > r$ 时 LoRA 贡献更大。

#### 为什么低秩增量就够了？

**"内在维度假设"（Intrinsic Dimensionality Hypothesis）：**

Li et al.（2018）发现，对很多 NLP 微调任务，优化轨迹的"有效维度"远小于参数空间维度。即：$\Delta W$ 在微调过程中是**近似低秩**的——大部分变化集中在少数方向。

直觉：预训练模型已经习得了语言通用特征。特定任务的微调只需要在这个基础上做**局部调整**（换个"头部"），不需要全局重塑整个参数空间。这个局部调整自然是低秩的。

**参数量对比（以 Llama-2 7B 为例）：**

```
注意力层权重矩阵：d = k = 4096
全参数微调单矩阵参数量：4096 × 4096 = 16.7M
LoRA (r=8)  单矩阵参数量：8×(4096+4096) = 65,536（减少 255×）

模型共有 32 层，每层 4 个注意力矩阵（Q、K、V、O）：
  全参数微调总量：7B 参数
  LoRA (r=8) 总量：约 33M 参数（< 0.5%）
```

#### 哪些层应用 LoRA？

优先对梯度幅度大（更新信号强）的矩阵应用：

| 矩阵 | 重要性 | 通常做法 |
|------|--------|---------|
| $W_Q$（Query） | 最高 | 必选 |
| $W_V$（Value） | 高 | 必选 |
| $W_K$（Key） | 中 | 可选 |
| $W_O$（Out projection） | 中 | 可选 |
| FFN 层 | 因任务而异 | 按需加 |

### LoRA vs QLoRA vs 全参数微调：如何选

| 方法 | 可训练参数 | 显存（7B模型） | 典型精度损失 | 适用场景 |
|------|----------|--------------|------------|---------|
| **全参数微调** | 100% | 80GB+ | 无 | 算力充足、追求最高精度 |
| **LoRA (r=8)** | ~0.5% | 16-24GB | < 1% | 大多数领域适配任务 |
| **QLoRA（4-bit + LoRA）** | ~0.5% | 8-12GB | 1-2% | 消费级 GPU / 低成本需求 |

**QLoRA（Dettmers et al. 2023）的原理：**

在 LoRA 基础上，将**基础模型权重量化为 4-bit**，然后在量化模型上添加 BF16 精度的 LoRA adapters：

```
基础模型权重：FP16 → NF4（Normal Float 4）
               ↓ 量化（减少 4× 显存）
LoRA Adapters：仍用 BF16 训练
               ↓ 梯度回传时按需解量化
优化器状态：    分页到 CPU（Paged AdamW）
```

NF4 是专为神经网络权重的正态分布设计的最优 4-bit 量化格式（最大化信息量），比普通 INT4 精度损失更小。

**实际决策：**

```
预算 = 1×A100（80GB）→ 全参数微调（< 20B 模型）或 LoRA（> 20B 模型）
预算 = 1×RTX 4090（24GB）→ LoRA（7-13B）或 QLoRA（65B）
预算 = 1×RTX 3090（24GB）→ QLoRA（7-13B 首选）
预算 = 消费级（< 16GB）→ QLoRA（7B 勉强可行）
```

**LoRA 的局限性：**
- 秩 $r$ 越小，可学习的任务特定知识越有限
- 对于需要大幅改变推理模式的任务（如数学推理），低秩可能不够
- LoRA adapters 在推理时需要合并（$W' = W_0 + BA$），不增加推理延迟，但增加了部署步骤

RAG 每次都要检索，Fine-tuning 知识在参数里直接用——两者的延迟差距有多大？对于 SLA 要求 500ms 以内的场景，RAG 能做到吗？

---

**🔧 工程师**

RAG 各阶段延迟分解（优化后的生产数据）：

```
完整 RAG 链路延迟（到首个 token）：

Query Rewrite（Haiku）：  100~200ms
并行检索（BM25 + ANN）：   50~150ms
Rerank（bge-reranker）：   50~100ms
LLM 生成（首个 token）：  200~500ms
─────────────────────────────────
总计：                    400~950ms
```

不加 Query Rewrite + 轻量 Rerank，可以压到 **300~600ms**。

**500ms 以内 SLA 的优化手段：**

| 优化点 | 节省时间 |
|--------|---------|
| 检索和 Rewrite 并行 | ~150ms |
| 热门查询结果缓存 | ~200ms（命中时） |
| 简单查询跳过 Query Rewrite | ~150ms |
| INT8 量化 Reranker | ~30ms |
| 流式输出（降低感知延迟） | 体感大幅提升 |
