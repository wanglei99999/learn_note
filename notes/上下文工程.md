# 上下文工程

上下文工程是在有限的上下文窗口中，选择、组织并注入与用户输入或任务高度相关的信息，从而让大语言模型能够在合理的边界内做出最佳推理和执行。最核心的一点是：**找到最相关的信息填充 LLM 的上下文窗口**。

## 一、Karpathy 的观点

Karpathy 在推特上的原话（[原文](https://x.com/karpathy/status/1937902205765607626)）：

> 大家一提到提示词，脑海里浮现的都是日常使用大模型时输入的简短任务指令。但在工业级大模型应用里，上下文工程是一门精细严谨的技术体系：精准填充上下文窗口，为模型下一步推理提供恰到好处的信息。
>
> 称其为科学，是因为做好这件事要统筹**任务说明、逻辑解释、少样本示例、检索增强生成（RAG）、多模态关联数据、工具调用、会话状态与历史记录、信息压缩**等诸多环节。上下文信息过少、格式不对，大模型就缺少有效支撑，输出效果达不到最优；信息冗余、无关内容过多，则会推高调用成本，反而削弱模型表现。把这套流程做到位，难度极高。
>
> 称其为艺术，是因为从业者需要凭借对大模型"推理特性"的经验直觉来把控整体设计。
>
> 除了上下文工程本身，一套完整的大模型应用还需要实现诸多配套能力：
> - 合理**拆解业务**问题，设计适配的控制流程
> - 精准**规划**、填充上下文窗口容量
> - 根据需求调度**匹配**对应能力规格的**大模型**
> - 搭建生成 - 校验**闭环**的**交互用户**界面与体验流程
> - 其余大量配套模块：内容**安全护栏**、**安全管控**、效果评测、**并行调度**、预加载等等
>
> 因此上下文工程，只是整套新兴复杂中间层软件中的一小部分；这层软件负责调度单次大模型调用，并串联各类能力，最终成型完整大模型应用。"ChatGPT 套壳"这种说法早已过时，而且存在根本性的认知偏差。

## 二、Anthropic 研究团队的观点

Anthropic 研究团队在《Effective Context Engineering for AI Agents》（[原文](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)）中也提出了类似观点。

### 从"提示工程"到"上下文工程"

上下文工程是提示工程（prompt engineering）的一次演进。过去我们主要专注于如何精心撰写指令；而现在，构建智能体的团队必须管理**在 LLM 推理过程中使用的那组最优 token（信息）**——这包括系统指令、工具、外部数据以及消息历史等所有内容。

### 核心挑战：上下文腐化（Context Rot）

研究表明存在一种被称为"上下文腐化"的现象——随着 token 数量增加，模型的准确率会下降。

由于 Transformer 架构会在所有 token 之间建立 O(n²) 的注意力关系，上下文越长，模型有限的"注意力预算"就被摊得越薄。不过这并不是一个"断崖"，而是"一种性能上的渐变，而非硬性的断崖"。

### 追求"最小的高信噪比 token 集合"

文章强调，要在以下各个组成部分中，寻找尽可能小的、信号密度最高的 token 集合：

- **系统提示（System prompts）**：在具体性和灵活性之间取得平衡——既要避免脆弱的 if-else 硬逻辑，也要避免含糊笼统的指导。
- **工具（Tools）**：保持工具集精简、功能不重叠，并配以清晰的参数定义。
- **示例（Examples）**：精心挑选多样化的"典范"示例，而不是堆砌详尽的边缘情况。

### 面向长周期任务的策略

对于长时间运行的任务，文章提出了三种技术：

1. **压缩（Compaction）**：对对话历史进行总结，然后用压缩后的上下文重新初始化。
2. **结构化笔记（Structured note-taking）**：让智能体维护可供随时查阅的外部记忆文件。
3. **多智能体架构（Multi-agent architectures）**：由专门的子智能体处理各自聚焦的任务，并只返回浓缩后的摘要。

### 运行时上下文检索：即时（Just-in-Time）方式

文章重点介绍了"即时（just-in-time）"的做法：智能体只维护轻量级的标识符（identifier），在需要时通过工具动态加载数据——这与人类的认知方式相似。

Claude Code 就是这种混合模式的典范：它既在启动时预加载 `CLAUDE.md` 文件，又保留了在运行时进行探索的能力。哪怕模型能力不断提升，"将上下文视为一种珍贵而有限的资源"这一理念仍将始终处于核心地位。
