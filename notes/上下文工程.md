
# 上下文工程 ：从"写好一句提示词"到"经营模型的认知环境"

> "多数 AI Agent 的失败，并非模型能力的失败，而是上下文工程的失败。"

这句话基本就是全文的中心思想。

如今它已经是开发 AI 智能体（AI Agent）必须掌握的核心技术领域。两件奠基性事件让这个技术走向大众视野：一是 Manus 在 2025 年 7 月公开分享，自己四次重构 Agent 开发框架沉淀出的实战经验；二是 Anthropic 在 2025 年 9 月发布了[《Effective Context Engineering for AI Agents》](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)这份文档。在这之后短短数月，文档里提到的各类上下文工程实践快速迭代成熟，被各大技术平台落地采用。

这篇文章尝试沿着一条**What、Why、How**的思路把它讲清楚：它到底是什么、我们为什么需要它、以及具体该怎么做；最后再看看Coding agent 的实测印证、2026 的工具生态、以及优秀Context Engineer 到底强在哪。

在展开之前，先给一张全文的导览图。它把后面所有章节压进一张图里：最外层是决定"模型能做什么"的 Harness（第七章），内层的上下文工程决定"模型看到什么"——由**上下文的组成**（① 第一章 What）、**四大操作**（② 写选压隔，第三章 How）、**底层技术栈**（③ 第五章）和贯穿全程的**上下文管理**（④）构成。看不懂细节没关系，读完全文再回看这张图即可。

## 一、What：什么是上下文工程

### 1.1 先厘清："上下文"到底指什么

很多人下意识以为，**上下文**就是聊天的历史记录。这个理解只对了一部分。**上下文（Context）是每一轮推理前投喂给模型的全部信息总和，范围远不止对话记录。** 它远比历史对话大得多。下面这张图，是一次推理时上下文窗口里可能装的东西：

![[context-composition-soft.svg]]

> 图：一次推理时，上下文窗口里可能装入的各类信号——系统提示、问题相关上下文、用户记忆、工具定义与输出、会话历史、结构化输出、用户输入，以及更多。窗口容量有限，需要不断裁剪与压缩。

单纯罗列各类信息没有意义。要落地管理，得先想清楚每一条上下文**从哪来、何时进窗口**（provenance & timing）——按这个来源与时机，可以干净地分成三类：

- **Config（配置层）**：运行前就由人预置、相对稳定的部分——System Prompt、Task Description（任务描述）、Few-shot Examples（少样本示例）、Output Schema（输出格式约束，即强制模型按 JSON/XML 等格式输出的结构定义），以及 **Tool Definitions（工具定义）、Skill Metadata（Skill 元数据）**。它们构成每次请求最稳定的前缀，会最先被 Prompt Cache 命中，也正是 Prompt Engineering 主要打磨的部分。
- **Retrieved（检索层）**：运行时从外部知识源检索或召回进来的——RAG 检索到的文档、Long-term Memory（长期记忆，跨会话的偏好与事实）里捞回的内容。
- **Runtime（交互层）**：本次运行中动态产生、按轮次追加的——User Input（用户输入）与会话历史、Tool Call / Tool Result（工具调用与返回，即历史追踪 Tool Traces）、Scratchpad（思考草稿，比如 Claude Code 开工前列的临时 TODO、Thinking 模型的"草稿本"），以及被触发后由 bash 载入的 SKILL.md 正文。

这样切有两个好处。其一，**"同型"的东西自然归在一起**——Tool Definitions 和 Skill Metadata 都是 capability descriptor（能力描述符），一并落在配置层；它们各自的调用/返回、正文载入则一并落在交互层。其二，这条"来源"轴和 1.5 的静/动分区、Prompt Cache 是**同一条线**：Config 是静态可缓存前缀，Retrieved 与 Runtime 是动态追加的部分（官方缓存顺序即 `tools → system → messages`）。

> 补一句边界：这是对 Anthropic 官方"静态 vs 动态"两分法的**细化**。官方把 System Prompt / 工具 / 示例等"预先塞入"的记为静态、把运行时检索记为动态；本文把动态一侧再拆成 Retrieved（检索来的）和 Runtime（跑出来的），以便区分二者——**拆成三类是本文所加，两分法才是官方原样**。

上下文因此是一个**多维、动态、服务于具体任务**的系统性概念。

### 1.2 那"上下文工程"是什么：一个好用的类比

Karpathy 提出过一个很贴切的类比：**整套大模型系统可以类比计算机操作系统。** 模型就像 CPU，上下文窗口就像 RAM（容量有限、需要精心调度），而**上下文工程，就是这个操作系统里的"内存管理器"**。它的职责不是把数据一股脑塞满 RAM，而是决定每一个"时钟周期"里，哪些信息该加载、哪些该换出、哪些该优先，好让整个系统跑得又快又准。

![[llm-os-claude.svg]]

> 图：LLM OS 示意——LLM 作 CPU，上下文窗口作 RAM，外接工具（计算器 / Python / 终端）、磁盘（文件系统 + embeddings）、浏览器、其他 LLM 等。据 Karpathy [《Software Is Changing (Again)》](https://www.youtube.com/watch?v=LCEmiRjPEtQ&t=620s) 中的构想重绘。

所以可以这样定义：**上下文工程是一门系统性学科，专注于设计并维护一个动态系统，在 agent 执行任务的每一步，为它智能地组装出最优的上下文组合。** 用 Karpathy 的原话：在上下文窗口中填入恰到好处的信息，为下一步推理做准备；用 Anthropic 的说法，是"找到最小的、信噪比最高的 token 集合"。

### 1.3 它和提示工程、RAG 是什么关系

 提示工程、RAG、上下文工程经常被混淆，**它们不是互斥关系，而是分属不同层级、互相协作的技术。**

- **提示工程是上下文工程的子集**，主要负责打磨 **Config 层**（尤其 System Prompt 那部分）；它面向单轮交互、粒度更细。
- **RAG 是充实 Retrieved 层的关键手段之一**：作用是从外部知识库中检索相关信息，作为 Retrieved 层的一部分填充到上下文窗口中。但上下文工程的范畴远大于 RAG，它不仅要考虑检索什么，还要把检索结果跟另外两类上下文动态组合起来。一个先进的系统，甚至可能在某些步骤完全不用 RAG，或在 RAG 失败后改用别的工具。

[Sourcegraph](https://sourcegraph.com/blog/context-engineering) 给出了一个很好区分的判断标准：**看你的改进来自"改措辞"还是"改接线"（rewording vs rewiring）**。只是换同义词、调语气，那还是提示工程；一旦你开始改"取哪些数据、按什么顺序、怎么重排、窗口满了淘汰谁"，就属于上下文工程。同时它把上下文工程的核心工作总结为四点：**取什么（fetch what）、何时取（when）、怎么压（compress）、何时丢（throw away）。**

Anthropic 有一张图把这层关系画得很清楚：

![提示工程 vs 上下文工程](https://www-cdn.anthropic.com/images/4zrzovbb/website/faa261102e46c7f090a2402a49000ffae18c5dd6-2292x1290.png)

> 图：左边是**提示工程**——单轮查询，上下文窗口里就 System prompt + User message，一问一答；右边是**面向 agent 的上下文工程**——从一大堆候选（Doc / Tool / Memory / 指令 / 领域知识……）里经过"精选（Curation）"，只把当下需要的组装进窗口，再随 Tool call → Tool result 不断循环；两侧的剪刀表示窗口有限、要不断裁剪。图源：Anthropic [《Effective Context Engineering for AI Agents》](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)。

### 1.4 换个视角：Context 的五层架构

如果说 1.1 的三类是按"来源"横切，那还有一种按"层级"纵切的分法（源自 [Taskade 的 Context Engineering 指南](https://www.taskade.com/blog/context-engineering)提出的五层 Context Stack），在工程上尤其清晰：

| 层 | 是什么 | 工程要点 |
|---|---|---|
| **System Context** | 身份层：System Prompt、角色定义、行为规范 | 要精简、稳定、可版本化。`AGENTS.md` / `CLAUDE.md` 就是它的工程化实现；HumanLayer 建议不超过 60 行，每行都要能追溯到一次具体失败 |
| **User Context** | 当前任务层：用户输入、当前会话历史 | 动态但体量不大，核心是会话历史管理（太长要截断/摘要） |
| **Retrieved Context** | 外部知识层：RAG 检索的文档、MCP 拉取的实时数据 | 最核心的一层，难在"检索什么、多少、怎么排" |
| **Tool Context** | 能力层：工具的名称、描述、schema | 工具描述本身就是 prompt，工具太多会导致上下文膨胀 |
| **Memory Context** | 长期记忆层：跨会话的偏好、约定、经验 | 不是越多越好，要有"遗忘策略" |

这五层不是独立的——`Retrieved Context` 太多，会挤压 `System Context` 的注意力。上下文工程的核心挑战，就是在这五层之间做**动态平衡**。

### 1.5 最终喂给模型的是什么：一次装配

前面讲的组成、三分类、五层都还偏抽象。落到一次真实推理：**上下文工程管理的，就是"喂给模型的那份输入"；而这份输入，是把上面各类信号序列化拼成的一条 token 流。** 这条流分成两个区：

- **静态前缀区**：System Prompt、工具定义、Skill 元数据等——相对稳定，放在最前面，可以被 **Prompt Caching** 缓存复用；
- **动态消息流**：会话历史、检索结果、工具调用与返回、当前输入——按轮次不断追加。

这里有一个容易被忽略、也最关键的点：**工具（tools）和技能（skills）不是"一个整块"塞进去，而是"定义在静区、产出在动区"的两段式。**

- **工具**：你传的工具定义（name + description + schema）落在**静态区**（大小取决于你挂了多少工具）；此外 API 还会**自动注入一段你看不到的"工具使用系统提示"**（Opus 4.8 约 290 token）。而真正的**工具调用（`tool_use`）与返回（`tool_result`）落在动态流**里，按轮累积。
- **技能**：只有**元数据（name + description，约 100 token/个）在启动时注入静态区**；被触发时，Claude 才用 `bash` 把 `SKILL.md` 正文读进来——也就是说**正文是以一次 `tool_result` 的形式进入动态流的**，脚本则只把运行输出带进来。

![[context-assembly-bar.svg]]

> 图：一次推理最终喂给模型的 token 流，按静态前缀区 / 动态消息流两分。工具与技能的"定义/元数据"常驻静态区并被缓存，"调用/返回/正文"在动态流按需生成——这既是它们省 token 的原理，也解释了为什么 Manus 强调"别在任务中途增删工具"（一改，后面所有缓存全失效，详见第三章）。图中数字为固定规格（Opus 4.8，源：[Anthropic tool-use](https://platform.claude.com/docs/en/agents-and-tools/tool-use/overview) 与 [Agent Skills](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview) 官方文档）；会话历史、RAG、工具返回等因场景而异，故不标数。

理解了这个"静/动分区"，后面第三章的很多做法就顺理成章：把稳定内容放前面吃缓存、把工具定义延迟加载（Tool Search，前置仅约 500 token）、把大块工具输出移出窗口只留索引——本质都是在经营这条 token 流的成本与信噪比。

## 二、Why：为什么需要这样一个新概念
前面我们已经对上下文工程的内容做了一定的解释，现在，我们要思考，为什么上下文工程会出现？
### 2.1 先看提示词工程遇到的三个问题

1. **上下文窗口有限**：不可能把整个 50 万行代码库、全部手册、所有历史对话都塞进一次请求。只塞相关的？但是"怎么知道哪些相关"本身就是一个难题。
2. **静态 vs 动态**：Prompt 是对话开始时写好的**静态**文本，但 agent 运行中要动态获取信息（实时天气、最新库存、第 3 步的中间结果）。静态 prompt 响应不了，得设计"按需注入"的机制。
3. **单点 vs 系统**：Prompt 只是一次输入，而 agent 需要的是一整套认知系统：工具（能执行动作）、记忆（能跨会话记住）、知识库（能访问训练外知识）、反馈循环（能从输出学习）。Prompt 只是这个系统的一个入口。

### 2.2 模型能力与上下文设计

当一个 agent 输出不及预期，根源无非两类：**模型能力不够**或**上下文没给对**。但在今天，基础模型的智能已经普遍越过某个阈值，答案通常指向后者，即上下文工程没有做好。你输出给模型的信息密度不够或者不正确，它就只能靠不确定的假设去填，甚至凭空幻觉。

Shopify CEO Tobi Lütke 那句被反复引用的话，正是这个意思：**"上下文工程，是让任务对 LLM 来说变得可解的艺术。"** 它把"模型能力不足"的归因，从"等待更强模型"变成了"改善上下文工程"。当 agent 做不好时，先别急着怪模型，先问自己：我有没有给它足够、且正确的上下文？

### 2.3 两个例子，看清它的必要性

**例子一：缺上下文的代价。** 设想你的 AI 助手收到一封很简单的邮件：

> 明天有空聚一下吗？

一个**上下文贫乏**的 agent，只看得到这一句、别无其它——代码可能完全正常，但回出来的东西又机械又没用：

> 我明天有空，请问约几点？

任务毫无推进。而一个**上下文充足**的 agent，在调用模型前会先动态组装：查日历（发现明天已满）、认出发件人是重要合作伙伴、翻过往邮件判断该用非正式语气、再挂上"发日历邀请"这个工具。于是它能回出：

> 嗨 Jim！我明天满了，周四上午方便吗？我先发了个暂定邀请，合适的话你确认下。

这里的"魔力"，不在更聪明的模型，而在**一个能为任务动态组装恰当上下文的系统**。

**例子二：全塞进去也不行。** 那把所有信息都给模型不就完了？当任务变长、变复杂，这种"简单累加"不仅失效，还会反噬。设想一个 coding agent 要在大型代码库里干好几天，如果把每一次指令、文件读取、编译报错、工具调用全部原样累积、每轮都完整传给模型，会发生一场**必然的失败级联**：

- **性能悄然下降**——早期已解决的语法错误之类的无关细节，不断稀释当前步骤的核心信息（上下文干扰）；
- **成本与延迟激增**——上下文线性膨胀，每次调用的 token 急剧上升；
- **最终崩溃**——撞上窗口上限，API 调用直接失败（上下文溢出），任务中断或因截断跑偏。

### 2.4 Context Rot：上下文会"腐坏"

**Context Rot（上下文腐化）**：即便窗口还没塞满，模型的表现也会随利用率上升而退化。有观察指出，**上下文用量超过 50% 后，模型对早期信息的注意力开始明显下滑；超过 80%，它甚至会"忘掉"系统指令里的关键规则**。

为什么会这样？三个原因：

- **注意力稀释**——Transformer 理论上能关注所有 token，但实践中存在"中间迷失"（lost in the middle，Liu 等人 2023 年的经典研究）：信息放开头或结尾用得最好，埋在中间就废了；
- **干扰增加**——上下文越长，无关噪声越多，模型得花更多认知资源过滤；
- **位置编码退化**——超长上下文里，位置编码分辨率下降，模型对顺序和关系的理解变差。

实验也印证了它的**非线性**：UC Berkeley 2024 年的研究发现，长文档问答里上下文从 4K 涨到 32K 时准确率略升，但从 32K 涨到 128K 反而下降——不是信息不够，是"太多导致注意力分散"。即便是缓解了"上下文焦虑"的新一代模型，在超长上下文里仍有注意力衰减。这也是为什么 Anthropic 推荐"完整重置"而非"无限堆叠"。

**所以核心结论只有一句：上下文是一种需要被主动管理的、有限的资源。** 它既不能缺，也不能滥。

## 三、How：具体怎么做

方法论之前，先看"会怎么坏"——理解了失效模式，才能对症下药。

### 3.1 先认识失效模式

在长程、复杂的 agent 系统里，除了最直观的"信息缺失（该检索的信息没有被没检索到）"，还有四种更隐蔽的问题：

- **上下文污染（Poisoning）**：幻觉混进了上下文，导致后续一错再错；
- **上下文干扰（Distraction）**：无关内容太多，模型注意力跑偏；
- **上下文混淆（Confusion）**：冗余、不相关的信息让输出跑偏；
- **上下文冲突（Clash）**：上下文里的信息自相矛盾（比如残留着过去的错误答案）。

再加上工程侧常见的**陈旧检索**（向量索引过期，把过时信息喂进来，对代码尤其致命，agent 会自信地调用一个废弃 API）、**中间迷失**、和**成本泄漏**（token 和工具调用没人盯着，悄悄超预算），就构成了一张相当完整的故障清单。解决办法通常不是"更少的上下文"，而是"更有针对性的上下文"，靠重排器强制一个硬上限，让模型拿到的是"刚好对的"信息，而不是"刚好有的"。

### 3.2 四大操作：写、选、压、隔

业界已经收敛出一套系统性的应对框架，可以概括为四个动作——**写入（Write）、选取（Select）、压缩（Compress）、隔离（Isolate）**。这套四分法出自 Lance Martin（LangChain）的[《Context Engineering for Agents》](https://rlancemartin.github.io/2025/06/23/context_engineering/)。逐个展开，并把当下最主流的技术和设计模式嵌进去。

#### 3.2.1 写入 Write —— 把信息持久到窗口之外

让 agent 把该记的东西写到上下文窗口**之外**，未来按需取回。分两种：

- **会话内写入（Session-level Write）**：在执行任务的过程中，把**任何对完成任务有用的信息**，如中间思考、计划、阶段性结论、工具的大块输出等随手写进会话内的**草稿纸（Scratchpad）**，轻量、临时，用于管理当前任务的复杂度。它的实现可以很简单：要么是一次工具调用（tool call），直接写进一个文件；要么就是 runtime 状态对象（state object）里的一个字段，在整个会话期间一直留着。Anthropic multi-agent researcher 里就有个很具体的例子——LeadResearcher 会先把整套方法想一遍、把计划存进 Memory，**因为一旦上下文超过 200,000 个 token 就会被截断，得先把计划保住**；
- **持久化写入（Persistent Write）**：把有长期价值的信息（用户偏好、关键事实）写进外部记忆系统（向量库、知识图谱、或结构化存储），实现跨会话的知识积累——ChatGPT、Cursor 就是这样在持续交互中"学习"的。

现阶段，按照记忆的理解角度不同，可以有两种对记忆的分类方法。一种按**生命周期**分类：短期（当前会话）、工作记忆（当前项目）、长期（跨项目经验），各有不同的存活时间和检索策略。另一种按**内容**分类（源自认知科学，Lance Martin 把它引进了 agent 语境）：**情景记忆（Episodic）**——过去把某件事做成的具体例子，落地成少样本示例；**程序性记忆（Procedural）**——"该怎么做"的规则本身，落地成 System Prompt 里的指令，或 `CLAUDE.md` / Cursor Rules 这类规则文件（Claude Code 正是把这类程序性记忆写进 `CLAUDE.md`，每次启动读回来）；**语义记忆（Semantic）**——关于世界的事实，落地成 RAG 检索到的知识。实际系统里两种叠着用：先认一条记忆是哪一类，再决定它该存多久、怎么取回。几条被验证的实践：

- **记忆棘轮原则(Ratchet Principle)**：只有经过验证的教训才写进长期记忆，且每条都能追溯到一次具体失败；配合定期审查、过期清理、冲突解决。
- **按需激活（on-demand activation）**：不是所有记忆每次都注入，而是按当前任务的关键词、文件路径、工具调用动态匹配——避免"30 条规则塞进 System Prompt 导致没一条被好好遵守"。
- **分类存储 + 分级检索**：用户偏好、项目配置、技术栈约定、历史教训分仓存放，按任务类型选择性加载（ChatGPT Memory、Cursor Rules 都是类似架构）。
- **Write-behind（延迟写入）**：不在对话中实时更新记忆，而在任务结束后异步提炼，避免频繁写入带来的上下文干扰。具体机制有两种——每轮结束后让 agent 自我反思、把教训生成为记忆（Reflexion），或周期性地从一批历史反馈里综合出记忆（Generative Agents 的做法）。

Anthropic 的建议是**把子 Agent 完整原始输出落地存文件，上下文只存文件索引，需要时直接读原始文件**，从根源消除上下文信息损耗。

#### 3.2.2 选取 Select —— 每次调用前，动态拉取最相关的信息
这是保证信噪比的关键，有三种方案：
1. **确定性选取（Deterministic Select）**：按固定规则加载，比如 Claude Code 启动就读 `CLAUDE.md`
2. **模型驱动选取（Model-driven Select）**：信息源太多时让模型自己筛
3. **检索式选取（Retrieval-based Select）**：目前最主流的方案，依靠相似度从知识库、记忆库调取内容；当下热门的多项技术都内嵌在这个环节中，细分三类实现思路。

**（1）路由（Routing）**
多领域智能体（agent）不会加载全部知识库、工具，比如计费类问题无需调用入职知识库；先完成查询分类，再分发到对应资源。路由共 4 种实现方式：
- LLM 路由（LLM-powered）：准确度高、可理解语义，但单次路由会增加推理耗时
- 分层路由（hierarchical）：主 agent 把任务拆分下发给专属子 agent（subagent）处理
- 规则路由（rule-based）：依靠关键词匹配，速度快但灵活性差，未预设的查询会直接失效
- 混合路由（hybrid）：结合上述几种的优势
节省下来的 token 可以在后续每一步流程持续复用。

**（2）渐进式披露 / Agent Skills**：
渐进式披露是一种「按需加载、不一次性把所有技能资料塞进模型上下文」的上下文节省技术，用来减少 token 消耗、降低模型推理负担、提升回答准确度。
1. 发现：启动时仅读取技能名称、描述，单条 skill 平均占用约 80 token。
2. 激活：查询匹配对应 skill 后，再读取完整指令，单条占用 275–8000 token。
3. 执行：真正运行该技能时，才加载它依赖的脚本文件。
Anthropic 2025 年 12 月推出的 Skills 格式（markdown + YAML），几周内被 OpenAI、Google、GitHub、Cursor 跟进适配。它一个核心用法是**身份管理（identity management）**：无需拆分多个独立 agent，单个基础 agent 可按需切换能力；日常使用基础身份，触发某类任务时加载对应 skill 的指令、约束、语气，任务结束后切回基础状态，Claude Code 就是该设计的典型案例（没有单独拆分 PDF 处理 agent、表格处理 agent）。更进一步，agent 可以把自身重复的处理逻辑封装成新 skill（对应 Claude Code 的 skill-creator 功能），形成「人工编写初始 skill → agent 从实操经验扩充技能库」的闭环。

![[agent-skills-progressive.svg]]

> 图：Agent Skills 的三级渐进披露。它在上下文工程里的位置就是「选取 Select」的一种——发现阶段只载入名称+描述（约 80 token），激活阶段才读完整指令（275–8000 token），执行阶段才加载依赖脚本。token 开销随层级递增，但只在真正需要时才付出。

**(3) Advanced RAG**
传统静态 RAG 流程「查询→向量搜索→内容注入→生成」，面对跨文档综合类问题（例如 “从本季度客户反馈中梳理出核心主题”）效果较差，因此演化成由 agent 管控的检索循环流程：
- **Agentic RAG**：传统 RAG 是固定流水线：用户发 1 次查询→检索一次→直接生成回答，检索质量差就直接输出劣质结果。Agentic RAG 把检索交给 Agent 自主管控：Agent 会自主判断检索效果：拿到检索内容后，判断信息够不够支撑作答；信息不足就改写检索关键词、发起多轮检索，反复迭代直到收集到足够信息，再输出答案；**用可循环的检索流程替代单向流水线**，适配跨文档、信息分散的复杂问题。。
- **Graph RAG**：普通向量检索只靠文本相似度匹配，很难梳理不同文档里实体之间的关联（比如客户、产品、投诉事件分散在多份文档，向量检索很难把三者串联推理）。Graph RAG 会先从全部语料提取实体、实体间关系，构建知识图谱；后续检索可以顺着关系链路做跨文档推理，解决向量检索不擅长的关联分析类需求。。
- **Self-RAG**：传统 RAG 不管问题需不需要外部资料，都会强制走检索流程，浪费算力、还容易引入无关内容。Self-RAG 让模型自主决策：先判断当前问题是否需要调取知识库内容，不需要就直接靠自身知识作答；需要检索时才触发检索，同时模型还会评判检索内容是否可靠，过滤无效、矛盾的检索片段。

**(4) 注意力感知放置（Attention-Aware Placement）**
我们在2.4提到了Transformer 存在**lost-in-the-middle**特性。工程上可以利用这个「首尾偏好」做优化：把最关键的指令放在 System Prompt 的开头和结尾、把 relevance 最高的文档放在检索结果的开头。

#### 3.2.3 压缩 Compress —— 用更少的 token 承载最核心的信号

窗口快满时的应对，有几套策略，可以按场景选：

| 压缩策略                             | 适用场景       | 信息损失          | 实现复杂度 |
| -------------------------------- | ---------- | ------------- | ----- |
| Sliding Window + Summary（滑窗+摘要）  | 多轮对话型任务    | 中（早期细节丢失）     | 低     |
| Hierarchical Summarization（层级摘要） | 研究/分析型长任务  | 低（层级保留结构）     | 中     |
| Checkpoint Reset（完整重置）           | 超长编码/重构任务  | 极低（结构化保存全部状态） | 高     |
| Selective Eviction（选择性淘汰）        | 工具密集型任务    | 中（仅保留最终结果）    | 中     |
| Token Budget Allocation（预算分配）    | 多 Agent 协作 | 可控（按优先级分配）    | 高     |

生产里最常见的是**滑动窗口 + 摘要**：最近几轮保持原始细节，更早的用 LLM 摘要掉（Claude Code 的 auto-compact 就是这么干；不过实践中它仍不完善，超长任务直接以最小上下文重启反而更稳）。工具的大输出（2000 行日志）只留头尾、完整版存文件按需读回（**修剪**的一种，代价是失去部分语境）。Manus 补了两个特别实用的细节：**最近几轮的工具调用保持原始格式**（维持模型的"节奏感"，否则会有微妙退化）；**报错轨迹不要压掉**（留着错误和堆栈能防止反复踩同一个坑）。Cognition AI 甚至专门 fine-tune 了一个"压缩大语言模型"来做这一步。压缩是有损的，所以要"先最大化召回、再迭代提精度"。

一个关键心法：**顶尖工程师不是等窗口快满才压缩，而是在任务开始前就规划好"上下文预算"**——System Prompt 占多少、工具描述占多少、检索预留多少、历史允许多少。就像内存管理，主动规划远优于被动回收。

#### 3.2.4 隔离 Isolate —— 在多条信息流之间设边界

前面三类优化，都是针对单条信息流内部做调整；隔离则从整体系统架构切入做设计：给不同subagent分配独立、干净的上下文，让它们分别处理海量原始资料，最终只把提炼完成的核心结论提交给主智能体。
Anthropic 有一个很贴切的观点：**搜索本身就是压缩，从海量资料里提炼关键结论**，这和 Ilya 提出的「压缩即智能」思路不谋而合。子智能体就相当于智能过滤器：它可以消耗数万 token 完成资料查阅、推导，最终仅输出一两千 token 的精简摘要传回主智能体。
主智能体不需要逐篇阅读原始文档，推理负担大幅降低，同时规避超长上下文容易出现的信息互相干扰、内容矛盾的问题，间接提升整体上下文的有效信息占比。多智能体协作、工具沙盒这类常用方案，底层都是隔离思路：Anthropic 多智能体调研工具的实测结果显示，多个上下文互相隔离的子智能体协同工作，效果远好于单智能体独自处理全部任务；HuggingFace 深度调研工具则把体量庞大的中间数据放进代码沙盒运算，避免这类数据挤占主智能体的上下文空间。

> **压缩 vs 隔离**，殊途同归。压缩针对单条信息流，提升单段内容本身的信息密度；隔离针对多条信息流，管控整套系统的复杂度，两种手段最终都是为了让上下文里有效内容更多，成熟的项目一般会同时使用两种方案。

除了这四大操作，还有几个常与之组合的**设计模式**值得单独记住：**自验证循环（Self-Verification Loop）**（agent完成一轮操作后自动开展核验，把出错内容补充进上下文，据此修正原有结果）、**结构化交接（Structured Handoff）**（拆分成长周期任务、跨会话执行时，整理包含任务目标、当前进度、待办事项、关键决策的交接文档，依托文档延续任务，远优于依赖模型自主记忆）、**上下文预检（Context Preflight Check）**（正式启动任务前，先核验现有信息是否充足、确认缺失内容，主动补齐信息缺口后再推进任务）。

### 3.3 别让工具描述吃掉半个窗口

工具是很容易被忽略的大额隐性开销，有必要单独拆解分析：每个工具的 JSON schema 结构定义本身就会消耗 token，一套复杂结构就要占用 500 以上 token；如果接入多个 MCP 服务、累计 90 多个工具，光是工具定义就要耗费 5 万 token，还没开始处理实际业务请求。

正因如此 OpenAI 建议单个智能体挂载工具数量不超过 20 个；从落地实践来看，工具数量超过 10 个，模型调用工具的准确率就会明显下滑。除了 token 消耗这类直接成本，工具管理还存在四类棘手问题：

- **描述质量问题**：模型依靠工具描述判断要不要调用该工具，但大部分 MCP 开发者撰写描述时是面向人类阅读的：描述写得太模糊，模型容易选错工具；描述写得太冗长，又会无谓消耗 token。
- **跨服务工具重复**：不同服务里会出现功能一致的工具，比如两个服务都自带搜索工具，系统没有做去重、调用优先级设定，模型只能随机挑选工具调用；
- **缺少版本管控**：服务端更新工具结构后，agent无法感知变更，缓存里留存的旧版工具描述会悄悄引发调用故障；
- **安全风险随工具数量上升**：每新增一个服务就新增一处攻击入口，工具返回的内容有可能夹带prompt注入攻击。

 Manus 还提到一个重要工程约束：不要在单轮任务迭代过程里新增、删减工具。工具定义放在上下文靠前的位置，一旦改动工具列表，后续所有步骤的 KV 缓存都会失效，整体推理速度会大幅下降。

顺带说明 MCP 相关背景，**MCP（Model Context Protocol）** 全称模型上下文协议，由 Anthropic 在 2024 年 11 月推出，目前交由 Linux 基金会下属的 Agentic AI 基金会维护，可以理解成 AI 工具领域的 USB 通用标准。在 MCP 出现前，对接每一类工具都需要单独开发适配代码；接入 MCP 后，工具提供方只要实现这套协议，客户端就能直接使用。但要注意：MCP 只统一了工具对接方式，并没有解决工具描述带来的上下文 token 开销问题。

### 3.4 各技术的选取是需要认真考虑的

不同技术在五个维度上各有取舍，横向对比如下，没有最好的，不同场景不同解：

| 技术 | 准确率 | 延迟 | Token 成本 | 可维护性 | 可靠性 |
|---|---|---|---|---|---|
| 渐进披露 / Skills | 能力集小时高，>100 个后因描述重叠误激活而下降 | 低（发现是文件读取，非 LLM 调用） | 静息低（17 个 skill 约 1700 token），会话中随激活累积 | 单个易维护，50+ 需治理 | 中，选错会级联，依赖发现层的选择精度 |
| 压缩 Compaction | 中，摘要有损 | 中，每次压缩需 LLM 调用（可周期性压缩摊薄） | 对长跑 agent 很低 | 需调参（留什么、几轮压一次、摘要多细） | 中，长任务好用，但早期关键细节被摘掉时差 |
| 路由 Routing | LLM 路由高、规则路由中 | LLM 路由加一次推理，规则近乎为零 | 下游持续省 | 规则要手动加新领域，LLM 自适应但难调试 | LLM 可能幻觉路由，需人工/默认兜底 |
| 检索进化 RAG | 最强项，可迭代改写、多策略 | 高，一问可能触发 3–5 轮 | 高，随复杂度上升 | 中，"为何选这个策略"比固定流水线难调 | 需护栏防简单问题也过度检索 |
| 工具管理 MCP | 取决于未解的描述质量 | 发现低，中途改工具使 KV-cache 失效 | 最大隐性成本，复杂 schema 500+/个，90 工具 50K+ | MCP 只统一接口，不管描述/版本 | 中，接口标准化≠工具本身可靠，且扩大攻击面 |

### 3.5 一个容易被问到的问题：提示工程去哪了？

你可能注意到，"写选压隔"讲的都是怎么管**动态流动**的信息，好像没提"怎么写 System Prompt"。这不是遗漏——由提示工程产出的那部分（也就是 1.1 的 **Config 层**，尤其 System Prompt），在这套框架里更像一种**相对静态的核心配置**，它会在"选取"这一步被加载进来。换句话说，提示工程没有消失，而是被安放进了一个更大的系统里。

### 3.6 落地：从哪儿下手

不必一次上全套，按你 agent 的形态定优先级：

- **长时间运行的任务** → 先上**压缩**（混合滑窗，配一个"探针"检查摘要有没有丢关键信息）；
- **跨多领域的 agent** → 先上**路由**（哪怕只是关键词规则，也能大幅减负，再考虑 LLM 分类）；
- **接了多个 MCP Server** → 连之前先**审一遍 token 成本**（数数工具 schema 在用户开口前吃掉多少 token，这数字通常比你以为的大）。

## 四、实战印证：编码 agent 是最好的"实验室"

多数讲上下文工程的文章用日历或宝可梦举例，但**代码是最好的实验场**——相关上下文是结构化的、检索问题定义清晰、失败几秒内就在测试里现形。

Sourcegraph 在 2026 年 2 月的 7.0 版里提出，代码智能平台是"开发者和 AI agent 共享的智能层"。它的 MCP Server 对任何 MCP 客户端（Claude Code、Cursor、Amp、Codex）暴露 13 个工具：跨库的关键词/语义搜索、符号解析与依赖追踪、跨库导航、提交/diff 历史、文件读取。底层是用 **SCIP**（一个开放的、基于 Protobuf 的代码智能协议）索引的代码图——**当结构存在时，直接返回定义和引用，而不是靠概率式文本检索**。问一个符号的定义，拿到的是"那一个定义所在的文件 + 三处调用点"，而不是"提到这个字符串的 50 个文件"。对严守 token 预算的 agent 来说，这就是"烧光窗口"和"完成任务"的区别。

它的 **CodeScaleBench 实测**（2026 年 3 月，370 个企业级任务，对比"本地 grep/文件读" vs "Sourcegraph MCP"）很能说明问题：

- 文件召回率 **0.127 → 0.277**，Precision@5 **0.140 → 0.478**，F1@5 **0.099 → 0.262**；
- 一个 Kubernetes 单体仓库任务，基线跑到 **2 小时超时**，用 MCP **89 秒**完成（得分 0.90/1.0）；
- 一个跨文件重构，基线 **96 次工具调用 / 84 分钟**，用 MCP **5 次调用 / 4.4 分钟**，且奖励翻倍。

Stripe 的 Minions agent 也接了 MCP 来"通过 Sourcegraph 搜索收集内部文档、工单、构建状态、代码情报"。Windsurf 的 Varun 早就点破过背后的道理：**给代码建索引 ≠ 上下文检索**——光有向量索引不够，得靠 grep / 文件搜索、知识图谱检索、再加一道重排的组合。对做代码多 agent 系统的人，结论是：**生产级的"检索支柱"很少是单个向量库，而是一个通过标准协议接入的、代码感知的检索层——有结构的地方用确定性结构，没结构的地方用语义搜索。**

## 五、2026 的工具与框架生态

一年下来，工具链比过去宽了不少，生产栈大多从四类里取用：

- **向量数据库**：Weaviate、Pinecone、Qdrant、Milvus、pgvector。差异主要在运维层面（混合检索、规模化过滤性能、重排器接入难易）。对多数团队，瓶颈不在数据库，而在喂给它的切块和 embedding 策略。
- **编排框架**：LangChain（最广用）、LlamaIndex（检索侧强）、DSPy（把 prompt 和检索作为一个整体优化）。它们都不是必需的——很多生产 agent 直接对着模型 SDK 写。
- **代码智能**：大/多仓代码库里，纯向量检索通常不够（代码有调用图、类型、跨库引用这些文本检索会丢掉的结构）。Sourcegraph MCP（SCIP 索引）是一个生产级答案，Amp 则默认接好了它。
- **记忆层**：mem0、Letta 是较显眼的框架，但多数团队仍在键值存储上自建一层薄记忆 + 一个摘要 pass，因为需求差异太大。

## 六、顶尖 Context Engineer 强在哪，以及别踩的坑

### 6.1 五种核心能力

技术和模式讲完了，但真正把这件事做到极致的还是人。观察业界最顶尖的 Context Engineer，他们有几个共同特征：

1. **信息流设计思维**——不是"写 prompt 的人"，而是"设计信息流的人"：这个任务每个决策点需要什么信息、从哪来、何时注入、何时移除、冲突了怎么定优先级。很像后端架构师设计数据流。
2. **注意力机制的工程直觉**——不是读论文，是靠大量实验建立的手感：核心指令放首尾、规则要精简一致（否则模型"选择性失明"）、利用率过高后新信息的"注意力收益"急剧下降、结构化格式（列表/表格/XML）比纯文本更易被遵循。
3. **"减法"思维**——普通工程师本能是"加"，顶尖的本能是"减"：每加一条先问"删了还能完成吗？会不会和已有信息冲突？注意力成本值不值？" HumanLayer 分享过一个经典案例——System Prompt 从 200 行精简到 60 行后，任务完成率反而**提升了 15%**。
4. **动态调试能力**——agent 表现差时，第一反应不是"换更强的模型"，而是导出模型实际看到的完整上下文、人工审查信息缺口、迭代实验建立"上下文变量 → 输出质量"的因果模型（Anthropic 内部把这叫 Context Profiling）。
5. **系统性思维**——把 agent 当"信息处理系统"而非"一次问答"：容量规划、并发控制、失败恢复、可观测性。所以最好的 Context Engineer 往往有分布式系统或操作系统背景。

一句话概括差别：普通工程师想的是"我该写什么"，顶尖工程师想的是"模型该看到什么、什么时候看到、看完怎么处理"。

### 6.2 四个反模式

- **上下文囤积症**：把所有可能相关的都塞进去"以防万一"→ Context Rot。正解是精选，宁可少给，不确定就先不给（模型会按需调工具获取）。
- **不管理会话历史**：历史无限增长、从不截断摘要 → 窗口被占满、模型"忘"掉最近指令。正解是定期压缩。
- **工具描述膨胀**：接 30 个 Server、300 个工具、30K token 描述 → 模型"选择困难"、选错或重复调用。正解是只启用当前需要的、用 Skill 渐进披露、定期删不常用的。
- **记忆不遗忘**：记忆文件越写越长、从不清理 → 规则太多导致注意力稀释、过时规则误导模型。正解是"棘轮原则" + 定期审查。

## 七、再上一层：从 Context 到 Harness

上下文工程解决的是"模型这一刻**看到什么**"，但它管不了"模型做完之后怎么办"——代码在哪运行、环境安不安全、出错怎么恢复、多个 agent 怎么协调、什么时候该停下来让人审核。这些属于 **Harness Engineering** 的范畴。放到全景里看，上下文工程其实是 Harness Engineering 的一个子集：

![[context-vs-harness.svg]]

> 图：上下文工程 ⊂ Harness Engineering。前者管"模型看到什么"（RAG、MCP 描述、记忆、压缩、隔离），失效表现为 Context Rot；后者管"模型能做什么"（沙箱、工具执行、子 agent 编排、人工审核），失效表现为早停与错误累积。

两者的分工大致是：

| 维度 | Context Engineering | Harness Engineering |
|---|---|---|
| 核心问题 | 模型**看到**什么 | 模型**能做**什么 |
| 关键组件 | RAG、MCP 描述、记忆、压缩 | 沙箱、Hook、工具执行、子 agent 编排 |
| 时间维度 | 持续动态组装 | 生命周期管理 |
| 失败模式 | Context Rot（信息过载） | 早停、错误累积 |
| 类比 | 办公桌上的资料 | 整个公司的运营体系 |

用一句流传的话说：**"Agent = Model + Harness；如果你不是模型，你就是 Harness。"** 上下文工程负责"让 AI 看到正确的世界"，但真正可靠的工程，是从让 AI 不只看到、还能行动、还能持续改进开始的。

---

## 收尾

我们的工作重心，已经不可逆转地从"如何找到那句完美的 prompt"，转向"如何设计一个能在每一步都为模型动态组装出恰当上下文的、健壮可靠的系统"——从"炼丹式地找 prompt"，回到扎实的系统工程上来。

说到底，无论是精巧的 prompt、RAG，还是 MCP，它们都指向同一个目标：**在模型做出决策之前，为它准备好一份恰到好处的上下文。** 模型的能力，不只取决于参数量，更取决于它每一刻所处的认知环境的质量。而把上下文当成一种珍贵、有限的资源来经营，正是这门手艺的核心。
